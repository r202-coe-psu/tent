# T-25 — Meal plan from occupancy × SOP ratio

## Summary

แผนอาหารรายวัน/รายมื้อ คำนวณวัตถุดิบอัตโนมัติจาก **occupancy จริง (T-06) × SOP ratio (T-31)**.
Kitchen staff เปิดฟอร์ม → ระบบดึงยอดผู้พักพิงที่ checked-in มา auto-fill → คำนวณ ingredient (ข้าวจาก SOP)
→ บันทึก audit trail (`calc_source`) → ส่งต่อเป็น input ให้ T-26 requisition.

**ตัวอย่างคำนวณ:** 100 คน × 150 g/คน/มื้อ = 15,000 g ข้าว

## What was built

### Occupancy → headcount (T-06 source) — CR-022
- `domain/occupancy.ts` — `deriveHeadcountFromOccupancy()` (pure): นับเฉพาะ evacuee ที่ `checked_in`
  - `total` = จำนวน checked-in ทั้งหมด
  - `halal` = `religion = 'muslim'`
  - `infant` = `special_needs` มี `'infant'`
  - `soft_food` = `special_needs` ∈ {`bedridden`, `chronic_illness`, `elderly`}
  - sub-count เป็นมิติตั้งฉาก (คนหนึ่งเป็นได้หลายกลุ่ม) → แต่ละช่อง ≤ total (บวกกันเกินได้)
- `useOccupancyHeadcount()` hook + live-query watch `evacuee`/`movement` → occupancy เปลี่ยน = re-calc

### Meal plan calc + audit trail — CR-021
- `domain/meal-calc.ts` — `calculateMealIngredients()`: headcount × ratio → recipes + `calc_source`
- `calc_source` = `{sop_profile_id, sop_profile_version, headcount_as_of}` persist ลง doc (source/as-of)

### Manual override — CR-022
- แก้ headcount ต่างจาก occupancy snapshot → **บังคับกรอก `override_reason`**
- ปุ่ม "ใช้ยอดล่าสุด" reset กลับเป็น occupancy; ตารางโชว์ ⚑ เมื่อมี override

### Handoff → T-26 — CR-022
- `toRequisitionInput(mealPlan)` แปลง recipe → stock item (`ingredient:rice` → `item:rice`, unit `g`),
  `qty_issued` เริ่ม 0 (T-26 กำหนดยอดจ่ายจริงตอน issue)

### Route capability guard — CR-024
- guard `requireKitchen()` = `system_admin` | `shelter_manager` | `kitchen_staff` (helper `hasStaffCapability`)
- ใช้กับ 3 route: `/back-office/kitchen`, `/kitchen/gas`, `/kitchen/production-board` แทน `requireAuth`
- UX gate ชั้น client; authorization จริงยังอยู่ที่ data layer

### Check-in stopgap (T-06 dependency)
- ปุ่ม "เช็คอิน" ใน `/back-office/evacuee-management` — สร้าง `movement` (check_in) + อัปเดต
  `current_stay.status → checked_in` เพื่อให้ occupancy (LIVE COUNT) ขยับจริงได้
- เพิ่มคอลัมน์ "สถานะ" ในตาราง (badge เขียว = อยู่ในศูนย์)
- **ไม่ใช่ T-06 เต็มรูป** (ยังไม่มี check-out/transfer/zone selector) — พอสำหรับ demo T-25

### Production-board (BOM) — coming soon
- เอาปุ่ม nav (BOM/Custom/จัดการ) ที่ชี้ไป production-board stub ออก กันหลงทาง
- เหลือ chip disabled "ฐานสูตร BOM (เร็วๆ นี้)" จนกว่า BOM editing จะเชื่อม doc จริง

### Out of scope (flagged)
- `getActiveSopProfile()` ใช้ `SHELTER_CODE` constant — pattern single-shelter ทั้งแอป;
  multi-shelter / `shelterStore` เป็นงาน cross-cutting แยก task

## Manual demo (แผนอาหาร 1 วัน → requisition handoff)

**เตรียมระบบ** (จาก repo root):
```bash
cp couchdb-session-example.ini couchdb-session.ini   # ครั้งแรกเท่านั้น
cp .env.example .env                                 # ครั้งแรกเท่านั้น
docker compose up -d
```
จาก `frontend/` (ต้องใช้ Node ≥ 22.13):
```bash
nvm use 22
pnpm dev
```

**ขั้นตอน demo:**

1. **Seed ข้อมูลตั้งต้น** — รัน seed script (สร้าง SOP profile 150 g + evacuee ตัวอย่าง)
   หรือ login แล้วลงทะเบียน + check-in evacuee ให้มียอด occupancy

2. **ดู LIVE COUNT** — เข้า `/back-office/kitchen`
   - badge "ผู้พักพิง (LIVE COUNT): N คน" ต้องแสดงยอด checked-in จริง

3. **สร้างแผนจาก occupancy** — กด "สร้างแผนอาหารใหม่"
   - ฟอร์ม auto-fill total/halal/soft_food/infant จาก occupancy
   - เลือกมื้อ → ดู preview "ข้าว N กรัม" (เทียบมือ: total × 150)
   - กด "สร้างแผน (draft)" → toast สำเร็จ + แถวขึ้นในตาราง (โชว์ "ข้าวสาร: N g")

4. **ทดสอบ override** — สร้างแผนอีกมื้อ แล้วแก้ total ให้ต่างจาก occupancy
   - ช่อง "เหตุผลที่แก้ยอด" โผล่ + ปุ่มบันทึก disable จนกว่าจะกรอก
   - บันทึก → ตารางโชว์ ⚑ พร้อมเหตุผล

5. **ยืนยันแผน** — กด "ยืนยันแผน" ที่แถว draft → เปลี่ยนเป็น "สำเร็จสมบูรณ์"

6. **ตรวจ audit trail** — เปิด Fauxton
   `http://localhost:5984/_utils/#database/shelter_sh001/_all_docs` → หา `meal_plan:*`
   - ต้องเห็น `calc_source` (sop_profile_id, version, headcount_as_of) + `override_reason` (ถ้า override)

7. **Re-calc on occupancy change** — check-in evacuee เพิ่มอีกหน้าต่าง →
   badge LIVE COUNT อัปเดตเอง; เปิดฟอร์มใหม่ยอด auto-fill สะท้อนค่าล่าสุด

8. **Handoff T-26** — `toRequisitionInput(mealPlan)` พร้อมใช้เป็น input ของ T-26
   (recipe → stock item). ทดสอบผ่าน unit test `meal-calc.test.ts` (T-26 UI เป็น task แยก)

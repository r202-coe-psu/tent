# Register Flow: Household Management

## 1. ปัญหาที่เจอ (Problem Statement)
ในการลงทะเบียนผู้ประสบภัย (Registration) ขั้นตอนที่ 2 (Stage 2: ข้อมูลผู้ประสบภัย) พบปัญหาในการตรวจสอบว่า **"มีข้อมูลครัวเรือน (Household) นี้อยู่ในระบบแล้วหรือไม่"** 
อิงจาก UI เดิม หากผู้ใช้งานพิมพ์ชื่อหรือรายละเอียดตัวหนังสือผิดเพี้ยนไปแม้แต่น้อย (Typo) จะส่งผลให้ระบบค้นหาไม่เจอ และอาจนำไปสู่การสร้างข้อมูลครัวเรือนที่ซ้ำซ้อนกันในระบบ

## 2. เหตุผลที่ควรเปลี่ยน (Reasons for Change)
1. **ลดความซ้ำซ้อนของข้อมูล (Data Deduplication):** การบังคับให้ค้นหาและเลือกจากฐานข้อมูลก่อน จะช่วยป้องกันการสร้าง Household ซ้ำซ้อนจากปัญหาการพิมพ์ผิด
2. **เพิ่มความแม่นยำในการเชื่อมโยงข้อมูล (Accurate Relationships):** การแยก Flow ชัดเจนจะช่วยให้ระบบผูกความสัมพันธ์ระหว่าง **ลูกบ้าน** และ **หัวหน้าบ้าน** เข้ากับ Household ได้อย่างถูกต้องเป็นระเบียบ
3. **ปรับปรุงประสบการณ์ผู้ใช้งาน (UX Improvement):** ลดความสับสนในขั้นตอนการกรอกข้อมูล โดยย้ายและแยกส่วนการจัดการที่อยู่ออกมาให้ชัดเจน ทำให้ผู้ใช้งานทำรายการได้ตรงตามวัตถุประสงค์ (Intent) มากขึ้น และลดภาระการกรอกฟอร์มที่อยู่ยาวๆ ซ้ำซ้อน หากบ้านหลังนั้นมีข้อมูลอยู่แล้ว

## 3. วิธีการเปลี่ยนแปลง (Proposed Implementation)
**การจัดการที่อยู่ (Household) ในหน้าจัดการครัวเรือน (Register):**
*   **ย้ายตำแหน่งฟอร์ม (Stage Move):** ย้ายช่องกรอกข้อมูลที่อยู่จากการจัดการ Household จากเดิมที่อยู่ใน **Stage 2 (ข้อมูลผู้ประสบภัย - Registration)** นำมาไว้ใน **Stage 3 (จัดการตั้งค่า หัวหน้าครอบครัว - Head of Household)** แทน
*   **ปรับรูปแบบการจัดการเป็น 2 Box (ทางเลือก):**
    1.  **Box 1: Search / Select Household (ค้นหาบ้านเดิม)**
        *   ให้ผู้ใช้ค้นหาสถานที่หรือบ้านเดิมจากระบบก่อน (อาจใช้ AutoComplete ค้นจากบ้านเลขที่หรือชื่อ) 
        *   เมื่อเลือกสถานที่ได้แล้ว ระบบจะผูก (sign) ให้บุคคลที่กำลังลงทะเบียนมีสถานะเป็น **"ลูกบ้าน"** (Member) ทันที
    2.  **Box 2: สร้างที่อยู่ใหม่ (Create New Household)**
        *   กรณีค้นหาไม่เจอ ให้ผู้ใช้กรอกข้อมูลที่อยู่ใหม่ตาม Flow ปกติ
        *   เมื่อบันทึกข้อมูล ระบบจะผูก (sign) ให้บุคคลที่กำลังลงทะเบียนเป็น **"หัวหน้าบ้าน"** (Head) ทันที

## 4. ไฟล์ที่ได้รับผลกระทบ (Affected Files)
1.  **UI Components ของขั้นตอนการลงทะเบียน:**
    *   `frontend/src/lib/features/register/ui/stage2-registration.svelte` (หรือ Component ที่ใช้คุม Stage 2): ต้องถอดฟอร์ม/Input ที่เกี่ยวข้องกับ Household ออกไป
    *   `frontend/src/lib/features/register/ui/stage3-head-of-household.svelte` (หรือ Component ที่ใช้คุม Stage 3): เพิ่ม Component สำหรับจัดการ Household โดยแบ่งเป็น 2 Box ชัดเจน (Search Box และ Create Form)
2.  **Schema Validation:**
    *   `frontend/src/lib/features/register/domain/schema.ts`: ต้องปรับปรุง Zod Schema ให้สอดคล้องกับ State ใหม่ (กรณีเลือกลูกบ้าน ตรวจสอบแค่ `household_id`, กรณีสร้างใหม่ ตรวจสอบข้อมูลที่อยู่ให้ครบถ้วน)
3.  **Backend API & State Management:**
    *   `frontend/src/routes/api/register/+server.ts`: ปรับ Logic ฝั่งหลังบ้านให้รองรับการทำงานของ Stage 3 ใน 2 รูปแบบ เพื่อบันทึกข้อมูลความสัมพันธ์ (ลูกบ้าน vs หัวหน้าบ้าน) ได้อย่างถูกต้อง
    *   การส่งต่อ State ข้อมูลระหว่าง Stage 2 ไปยัง Stage 3 ใน Store ของ Svelte

## 5. สิ่งที่ต้องปรับปรุงใน Spec (Spec Updates)

1.  **แก้ไข Description:** ระบุให้ชัดเจนว่าการตั้งค่าถูกย้ายไปอยู่ Stage 3
    ```diff
    - **Description:** สร้าง "ครัวเรือน" เป็น optional grouping เหนือ person record...
    + **Description:** สร้าง "ครัวเรือน" เป็น optional grouping เหนือ person record... โดยการจัดการและตั้งค่า Household จะแยกมาอยู่ใน **Stage 3 (จัดการตั้งค่าหัวหน้าครอบครัว)** อย่างชัดเจน เพื่อลดความสับสนกับฟอร์มข้อมูลบุคคลใน Stage 2
    ```

2.  **แก้ไข Flow Path A (สร้าง ณ จุดรับ):** เพิ่มทางเลือกย่อย (Search Existing / Create New)
    ```diff
    - **Flow — Path A (สร้าง ณ จุดรับ):**
    - 1. VOL กรอก head + สมาชิก + emergency contact ของ head
    - 2. ระบบ validate → ออก Shelter ID + QR (T-05) ทันที
    - 3. ดำเนินต่อที่ check-in (T-06)
    + **Flow — Path A (สร้าง ณ จุดรับ) — ทำในขั้นตอน Stage 3:**
    + 1. VOL ตรวจสอบและจัดการ household โดยแบ่งเป็น 2 ทางเลือก:
    +    - **A.1 ค้นหาบ้านเดิม (Search Existing):** ค้นหาสถานที่/บ้านเลขที่จากระบบ หากพบ ให้เลือกเพื่อผูกบุคคลที่ลงทะเบียนนี้เป็น **"ลูกบ้าน"**
    +    - **A.2 สร้างบ้านใหม่ (Create New):** หากไม่พบ ให้สร้างฟอร์มที่อยู่ใหม่ และระบบจะผูกบุคคลนี้เป็น **"หัวหน้าบ้าน"** อัตโนมัติ (พร้อมกรอก emergency contact)
    + 2. ระบบ validate → ออก Shelter ID + QR (T-05) ทันที (เฉพาะกรณีสร้างบ้านใหม่)
    + 3. ดำเนินต่อที่ check-in (T-06)
    ```

3.  **ปรับปรุง Definition of Done (DoD):** เพิ่มเกณฑ์เรื่อง 2 Flow ชัดเจน และการ Assign Role
    ```diff
      **Definition of Done:**
    + - API + UI ของระบบลงทะเบียน (Stage 3) มีการแบ่งแยก Flow ค้นหาที่อยู่เดิม (รับบทลูกบ้าน) และสร้างที่อยู่ใหม่ (รับบทหัวหน้าบ้าน) อย่างชัดเจน
    + - ระบบสามารถ Assign Role สมาชิก (ลูกบ้าน/หัวหน้าบ้าน) ให้ตรงตามเงื่อนไขทางเลือกโดยอัตโนมัติ
      - API + UI สร้าง/แก้ไข household, เพิ่ม-ถอดสมาชิก, ตั้งและเปลี่ยน head ได้ (head ต้องเป็น person ที่มี required fields ตาม FR-5)
    ```

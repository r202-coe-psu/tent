# Smart Shelter — Glossary & Technical Terms

เอกสารนี้รวบรวมคำอธิบายตัวย่อ ศัพท์เทคนิค และรหัสอ้างอิงทั้งหมดที่ใช้ในชุดเอกสาร [Task Breakdown](_index.md) และระบบการทำงานของโครงการ **Smart Shelter** (ระบบบริหารจัดการศูนย์พักพิงอัจฉริยะ) เพื่อให้ผู้พัฒนาใหม่หรือทีมงานภายนอกเข้าใจทิศทางและสถาปัตยกรรมของโครงการได้ทันที

---

## 1. รหัสอ้างอิงเอกสารและกระบวนการพัฒนา (Process & Planning Acronyms)

| ตัวย่อ / รหัส | คำเต็ม / ความหมาย | คำอธิบายและบริบทในโครงการ |
| :--- | :--- | :--- |
| **FD** | **Functional Design** | เอกสารการออกแบบฟังก์ชันหรือข้อกำหนดระบบย่อย เช่น [FD-12](11-famsearch.md#L12) (สิทธิส่วนบุคคลในระบบค้นหาครอบครัว), [FD-14](10-eoc.md#L12) (สถาปัตยกรรม EOC API), [FD-15](_index.md#L6) (การแตกงานรายโมดูล) |
| **FR** | **Functional Requirement** | ข้อกำหนดเชิงฟังก์ชันที่ระบุว่าระบบต้องสามารถทำอะไรได้บ้าง (เช่น FR-21 ถึง FR-56) |
| **NFR** | **Non-Functional Requirement** | ข้อกำหนดด้านประสิทธิภาพ ความปลอดภัย สถาปัตยกรรม หรือคุณสมบัติอื่นๆ ที่ไม่เกี่ยวข้องกับฟังก์ชันตรงๆ (เช่น NFR-18/21 เรื่องประสิทธิภาพของแบบจำลองการอ่านข้อมูล หรือ NFR-25 เรื่องการ Deploy ระบบ) |
| **K-13** | **Kickoff Meeting Reference** | รหัสอ้างอิงสรุปการประชุมเปิดตัวโครงการ (Kickoff Meeting) สำหรับการตกลงจัดทีมรับผิดชอบโมดูลย่อย (ทีม D1–D4) |
| **P-01, P-02, P-03** | **Pre-production Design** | งานออกแบบทางเทคนิคจากบริษัทคู่สัญญา (บริษัท เน็ท) เพื่อเป็นแนวทางและพิมพ์เขียวสำหรับการเริ่มพัฒนาซอฟต์แวร์จริงในแต่ละเฟส |
| **R2, R3, R4** | **Release 2, 3, 4** | ลำดับและเฟสย่อยในการพัฒนาโครงการ:<br>• **R2 (Foundation)**: วางฐานระบบสิทธิ์ สกีมาข้อมูลหลัก ครัว และโมดูลพื้นฐาน<br>• **R3 (Operations)**: ระบบจัดการหน้างาน เช่น แผนอาหาร การคำนวณทรัพยากร และอาสาสมัคร<br>• **R4 (Integration)**: ระบบเชื่อมต่อ เช่น แพลตฟอร์มค้นหาครอบครัว, EOC และ Open API |
| **T-xx** | **Task ID** | รหัสของงานพัฒนาหลักแต่ละชิ้นที่ระบุในแผนการปฏิบัติงานและเส้นวิกฤต เช่น T-01 ถึง T-46 |
| **Q-xx** | **Quality & Support Task ID** | รหัสงานพัฒนาในส่วนการสนับสนุน เช่น การ Deploy อัตโนมัติ (Deployment Automation), การซัพพอร์ตระบบหลังเริ่มใช้งานจริง (Support Buffer), การเขียนคู่มือสำหรับผู้ใช้ |
| **MD** | **Man-Days** | วัน-คน (หน่วยประเมินกำลังแรงงานที่ต้องใช้ในงานพัฒนาซอฟต์แวร์ โดย 1 MD หมายถึง แรงงานของนักพัฒนา 1 คนที่ทำงานเต็มเวลา 1 วัน) |
| **Raw MD** | **Raw Man-Days** | จำนวนแรงงานดิบที่คาดคะเนในตอนแรก ก่อนนำตัวคูณลดทอนระยะเวลามาคำนวณ |
| **AIx** | **AI Multiplier** | ตัวคูณลดทอนเวลาจากการนำระบบ AI (เช่น GitHub Copilot) เข้ามาช่วยเขียนโค้ดเพื่อประหยัดเวลา (แสดงในรูปของการหาร เช่น ÷1.25, ÷1.4, ÷1.6) |
| **Adj MD** | **Adjusted Man-Days** | จำนวนแรงงานสุทธิหลังจากการคำนวณด้วยตัวคูณ AI เรียบร้อยแล้ว (`Adj MD = Raw MD / AIx`) ใช้เป็นเกณฑ์ในการคำนวณระยะเวลาใน Gantt Chart |
| **WD** | **Working Days** | วันทำการจริงในการพัฒนาโครงการ |
| **UAT** | **User Acceptance Testing** | การทดสอบขั้นสุดท้ายเพื่อยอมรับระบบโดยผู้ใช้งานจริง เพื่อตรวจสอบความถูกต้องก่อนเริ่มใช้งานจริงในพื้นที่อพยพ |
| **Spike** | Spike research | งานศึกษา/วิจัยเชิงเทคนิคสั้นๆ เพื่อพิสูจน์แนวคิดหรือเทคโนโลยีที่ไม่มีความแน่นอนก่อนเริ่มลงมือเขียนโค้ดจริง (เช่น T-17 Spike เรื่องการผูกโยงคลังสินค้ากับครัว) |
| **Walking Skeleton** | Walking skeleton code | โครงสร้างระบบเบื้องต้นที่ทำงานได้จริงในแนวดิ่ง (End-to-End) ตั้งแต่ฐานข้อมูล ระบบสิทธิ์ และหน้าจอหลัก เพื่อให้ทีมพัฒนานำไปเป็นต้นแบบในการเริ่มงานสร้างสรรค์ส่วนอื่นๆ |

## 2. ระบบงานและข้อกำหนดเฉพาะของศูนย์พักพิง (Business/Domain Terms)

| คำศัพท์ / ตัวย่อ | ความหมาย | บทบาทและขอบเขตในโครงการ |
| :--- | :--- | :--- |
| **Smart Shelter** | ระบบบริหารจัดการศูนย์พักพิงอัจฉริยะ | แพลตฟอร์มดิจิทัลที่มุ่งเน้นการแก้ปัญหาและช่วยบริหารจัดการสิ่งต่างๆ ภายในศูนย์พักพิงอพยพจำนวน 11 ศูนย์ในจังหวัดสงขลา |
| **Hat Yai ROD** | **Hat Yai Regional Operations Center** | ศูนย์ปฏิบัติการภาพรวมระดับภูมิภาคของหาดใหญ่ ซึ่งรับผิดชอบในการเตือนภัยก่อนน้ำท่วมและบริหารจัดการภาพรวมระดับจังหวัด (เป็นผู้ดึงข้อมูลจาก Smart Shelter ผ่าน Open API) |
| **EOC** | **Emergency Operations Center** | ศูนย์ปฏิบัติการฉุกเฉิน / ศูนย์สั่งการเหตุการณ์ภัยพิบัติ ในโครงการนี้ใช้สื่อถึงการรวมข้อมูลและสรุปภาพรวมจากทุกศูนย์อพยพเพื่อส่งต่อให้ระดับจังหวัดเข้าถึงได้ทันที |
| **One Data Platform** | แพลตฟอร์มศูนย์กลางข้อมูลร่วม | ฐานข้อมูลกลางที่ Hat Yai ROD และผู้บริหารระดับภูมิภาคเข้าดึงข้อมูลไปใช้ประโยชน์ในการรับมือสถานการณ์ฉุกเฉิน |
| **ปภ.** | กรมป้องกันและบรรเทาสาธารณภัย | หน่วยงานรัฐหลักที่เป็นเจ้าภาพควบคุมการดูแลภัยพิบัติและการจัดตั้งศูนย์อพยพพักพิง |
| **SOP** | **Standard Operating Procedure** | ขั้นตอนการปฏิบัติงานมาตรฐาน ในระบบนี้คือสัดส่วนความต้องการทรัพยากร/อาหารที่คำนวณตามจำนวนยอดผู้ลี้ภัยจริง (เช่น อัตราส่วนอาหารต่อน้ำหนัก/อายุ/โรคประจำตัว) |
| **PII** | **Personally Identifiable Information** | ข้อมูลส่วนบุคคลที่สามารถระบุตัวตนจริงของผู้อพยพได้ (เช่น ชื่อ เลขบัตรประจำตัวประชาชน เบอร์โทรศัพท์) |
| **RoPA** | **Record of Processing Activities** | บันทึกกิจกรรมการประมวลผลข้อมูลส่วนบุคคล เพื่อยืนยันว่าการประมวลผลข้อมูลส่วนบุคคลของผู้อพยพสอดคล้องตามหลักกฎหมายความเป็นส่วนตัว (PDPA) |
| **Consent / Opt-out** | ความยินยอม / การขอยกเลิกสิทธิ์ | สิทธิ์ที่ระบบมอบให้ผู้อพยพในการเลือกว่ายินยอมที่จะให้เผยแพร่ประวัติของตนลงระบบค้นหาครอบครัวหรือไม่เพื่อความคุ้มครองและความปลอดภัย |
| **Anti-enumeration** | ระบบป้องกันการเก็บกวาดข้อมูล | มาตรการความปลอดภัยในการออกแบบระบบค้นหาสาธารณะ เพื่อป้องกันไม่ให้บุคคลภายนอกป้อนข้อมูลเดาสุ่มรหัสหรือรายชื่อเพื่อดึงสืบค้นข้อมูลประชากรทั้งหมดออกมาได้ |
| **Donor Pre-declaration** | การลงทะเบียนของบริจาคล่วงหน้า | ระบบที่ให้ผู้ใจบุญแจ้งรายการของที่จะนำมาบริจาคผ่านเว็บ/คิวอาร์โค้ดก่อนเดินทางมาถึงหน้างาน เพื่อความสะดวกรวดเร็วในการบริหารพื้นที่คลังสินค้าและโควตาการรับ |

## 3. สถาปัตยกรรมระบบและฐานข้อมูล (Technical & Architecture Terms)

คำศัพท์อธิบายเชิงลึกเกี่ยวกับสถาปัตยกรรมทางเทคนิคของฐานข้อมูลและการ failover ภายใต้สถานการณ์เครือข่ายไม่เสถียร (เช่น น้ำท่วม ไฟดับ อินเทอร์เน็ตขาดหาย):

| คำศัพท์ / ตัวย่อ | ความหมาย | บทบาทและประเด็นเชิงวิเคราะห์ในระบบ |
| :--- | :--- | :--- |
| **CouchDB** | ระบบฐานข้อมูล NoSQL แบบ Document | ฐานข้อมูลฝั่ง Server หลักของโครงการ โดย Central CouchDB เป็น source of truth ปกติ ส่วน LAN Edge CouchDB เป็น fallback replica เฉพาะช่วง WAN/Central ล่ม |
| **Remote-First** | แนวคิดเขียนไปเซิร์ฟเวอร์ก่อน | client เขียนไป active endpoint โดยตรง (Central ก่อน, Edge เมื่อ outage); **ไม่มี** local client DB เป็น write path ([CR-033](../changes/CR-033-remote-first-architecture-program-index.md)) |
| **PouchDB** | (legacy / out of scope) | เคยเป็นสมมติฐาน local-first — **ตัดออกจาก client write path ตาม CR-033**; เก็บคำนี้ใน glossary เพื่อกันสับสนเอกสารเก่าเท่านั้น |
| **Offline-First** | (legacy / out of scope) | สมมติฐานเดิมที่เขียน local ก่อนแล้ว sync — **supersede โดย Remote-First (CR-033)**; full offline sync ทุก module = non-goal |
| **Active endpoint** | ปลายทางเขียนที่ active ครั้งละหนึ่ง | Central เป็นค่าเริ่มต้น; สลับไป LAN Edge เฉพาะ outage; ห้าม active Central+Edge พร้อมกัน; เมื่อไม่มี endpoint → disconnected status-only |
| **Disconnected (status-only)** | โหมดขาดการเชื่อมต่อ | แสดง banner cannot-connect; ไม่มี read-only local cache และไม่มี local write queue; user บังคับ retry ได้หลัง auto-retry 3 ครั้ง |
| **Multi-Master Replication** | การคัดลอกข้อมูลแบบหลายจุดเขียนพร้อมกัน | ความสามารถของ CouchDB ที่รองรับหลาย replica เขียนได้ แต่ topology โครงการนี้ควบคุม client ให้มี active endpoint ครั้งละหนึ่งตัว ไม่รัน write ไป Central และ Edge พร้อมกัน |
| **Conflict & Resolution** | ความขัดแย้งของข้อมูลและการจัดการ | สถานการณ์ที่อุปกรณ์/endpoint ตั้งแต่ 2 จุดแก้เอกสารเดียวกันระหว่างfailover — ระบบมีกลไก deterministic winner และทดสอบ failback ไม่สร้าง duplicate |
| **Event-Sourcing / Event Ledger** | การบันทึกข้อมูลแบบประวัติเหตุการณ์ | สถาปัตยกรรมระบบที่ไม่บันทึกยอดรวมปลายทาง แต่บันทึกเฉพาะข้อมูลนำเข้า/นำออกต่อเนื่องกัน (Append-only) วิธีนี้ช่วยลดปัญหาการชนกันของยอดคงคลัง (Conflicts) |
| **RBAC** | **Role-Based Access Control** | การระบุและควบคุมการเข้าถึงระบบข้อมูลของผู้ใช้งานตามตำแหน่งหน้าที่ (Roles) เช่น ผู้ดูแลระบบ (Admin) หรือหัวหน้าคลังสินค้า (Warehouse Manager) |
| **Field-level permissions** | สิทธิ์การเข้าถึงข้อมูลระดับฟิลด์ | ข้อกำหนดการจำกัดการมองเห็นข้อมูลเฉพาะบางฟิลด์ในแฟ้มข้อมูลชุดเดียวกัน เช่น พนักงานทั่วไปมองเห็นชื่อผู้ลี้ภัยได้แต่ไม่สามารถมองเห็นข้อมูลโรคประจำตัวได้ |
| **_security document** | เอกสารระบุสิทธิ์ความปลอดภัยใน DB | การกำหนดความปลอดภัยของฐานข้อมูลในระดับฐานรากของ CouchDB เพื่อระบุให้ชัดเจนว่าระบบหรือผู้ใช้งานที่มี roles ใดบ้างที่จะเข้าถึงฐานข้อมูลชุดนี้ได้ |
| **validate_doc_update** | ฟังก์ชันสแกนข้อมูลก่อนเขียนลง DB | ฟังก์ชันการเขียนโค้ดตรวจสอบบนเซิร์ฟเวอร์ CouchDB เพื่อทำหน้าที่ตรวจประเมินรูปแบบสิทธิ์และข้อมูลที่ถูกเขียนเข้ามาว่าได้รับอนุญาตและผ่านเกณฑ์ที่ยอมรับได้หรือไม่ |
| **Backend proxy / BFF** | ตัวกลางจัดระเบียบข้อมูล | บริการตัวกลาง (รวม `/api/*`) ที่รับคำร้องขอ คัดกรอง จัดรูป และตรวจสอบความปลอดภัยก่อนส่งต่อไป CouchDB — path หลักของ remote-first |
| **DB-per-scope (Pattern 1)** | สถาปัตยกรรมแยกฐานข้อมูลตามศูนย์ | แนวทางการออกแบบโดยให้แต่ละศูนย์ย่อยใช้ Database ของตัวเอง เพื่อความรวดเร็วในการซิงก์ข้อมูล ประหยัดแบนด์วิดท์ และป้องกันไม่ให้เกิดความล่าช้าจากฐานข้อมูลขนาดใหญ่ |
| **Replication Topology** | สถาปัตยกรรมการไหลของข้อมูล | โครงสร้าง remote-first: client เขียน Central เมื่อ reachable, switch ไป LAN Edge เฉพาะช่วง WAN/Central outage, disconnected status-only เมื่อไม่มี remote, และ failback กลับ Central เมื่อฟื้นตัว |
| **Edge Computer** | เซิร์ฟเวอร์ท้องถิ่นประจำศูนย์พักพิง | เซิร์ฟเวอร์ทางกายภาพที่ติดตั้งอยู่ที่หน้างานของแต่ละศูนย์เพื่อเป็น LAN continuity/fallback replica ระหว่าง WAN/Central ล่ม ไม่ใช่ normal client hub และไม่ให้ session ใช้แทน Central API ได้ |
| **Soft-edge hub** | แท็บเล็ต fallback replica ชั่วคราว | สำหรับศูนย์พักพิงที่ไม่มีเซิร์ฟเวอร์ Edge จริง อาจกำหนดแท็บเล็ตหลักเป็น fallback replica เฉพาะช่วง outage ได้ แต่ default คือ Central เมื่อ online และ disconnected status-only เมื่อไม่มี remote |
| **Peer mesh** | โครงสร้างเครือข่ายซิงก์ไร้ตัวกลาง | ไม่ใช่ normal path ของโครงการ; topology ที่รองรับคือ Central + Edge fallback ตาม CR-033 |
| **Filtered replication** | การซิงก์ข้อมูลแบบคัดกรองเฉพาะหัวข้อ | การสตรีมข้อมูลเฉพาะชุดที่จำเป็นระหว่าง Central↔Edge (เช่น `_users` filter) เพื่อลดแบนด์วิดท์ — ไม่ใช่ client↔Pouch sync |
| **API Contract Freeze** | ข้อตกลงข้อมูล API คงที่ | ข้อตกลงในการพัฒนาที่กำหนดไม่ให้มีการเปลี่ยนโครงสร้างของ API ชั่วคราว เพื่อให้นักพัฒนาฝั่งหน้าบ้านและหลังบ้านสามารถแยกย้ายกันเขียนโค้ดได้โดยไม่ติดขัดซึ่งกันและกัน |
| **App-level event channel** | กลไก live-update มาตรฐาน | ช่องทาง invalidate/refresh ที่แอปใช้หลัง write สำเร็จหรือรับ notification จาก remote — แทน Pouch changes feed (CR-033 locked decision B) |

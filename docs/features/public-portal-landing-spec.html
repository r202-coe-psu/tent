Feature Flow Spec

Public Portal — Landing Page (ระบบข้อมูลสาธารณะ)

Back Draft v0.3 2026-06-22

Summary

หน้า Landing Page ของ Public Portal — จุดเข้าถึงระบบสาธารณะโดยไม่ต้อง login สำหรับประชาชนทั่วไป, ญาติผู้ประสบภัย, ผู้บริจาค และอาสาสมัคร ทำหน้าที่เป็น single entry point ที่รวม: search hero พร้อม real-time metrics, ประกาศฉุกเฉิน, ทางลัดบริการด่วน, FAQ และช่องทางติดต่อฉุกเฉิน

หน้านี้เป็น Public / No-Login Tier — ต้องไม่มีข้อมูลส่วนบุคคลใด ๆ ปรากฏ. ข้อมูลที่แสดงเป็น aggregate/static เท่านั้น. รายละเอียด trust boundary ดูที่ Public Tier Flow Spec
ออกแบบให้ใช้งานได้บนมือถือในสภาวะ network ไม่เสถียร — load เร็ว, ไม่พึ่ง JS สำหรับ content หลัก, รองรับ TH/EN (EN toggle)
v0.2 (2026-06-21): ปรับตาม mockup รอบใหม่ — (1) สลับลำดับ ให้ Search Hero อยู่บนสุด, Alert Banner ย้ายลงมาใต้ hero; (2) เพิ่ม Real-Time Metrics panel ใน hero; (3) quick service cards เพิ่ม badge; (4) ปรับ copy ของ help center / emergency contacts เป็นโทน EOC
v0.3 (2026-06-22) — decisions เคาะแล้ว: (1) ปิด (disable) ส่วน Transparency และ Volunteer ไว้ก่อน — nav links + volunteer quick-service card + volunteer metric; (2) FAQ เป็น dynamic ต้องมี setup ใน EOC (OP-1); (3) LINE OA / FB URL เป็น setup ใน EOC ไม่แสดงผลตอนนี้ (OP-2); (4) alert level 4 บน mobile = sticky top (OP-4); (5) ลิงก์แผนที่ = internal /shelters (OP-5); (6) metric "ผู้ประสบภัยปลอดภัย" = occupancy รวม — approved ให้แสดงต่อ public ได้ (OP-6, 2026-06-22); (7) metrics polling = ทุก 10 นาที, stale-data threshold = 30 นาที (OP-7)

Phase R3 — MVP Public Portal

Layout Overview (top → bottom)

  1. Global Navigation Bar — logo, nav links, language toggle
  2. Search Hero + Real-Time Metrics — dark gradient hero card: ซ้าย = search ญาติ, ขวา = real-time metrics panel
  3. Emergency Alert Banner — แสดงเฉพาะเมื่อมี active alert level ≥ 1; ซ่อนเมื่อไม่มี
  4. Quick Services Grid — 3 card ที่ active (ผู้ประสบภัย / บริจาค / ค้นหาญาติ; volunteer disabled v0.3) แต่ละ card มี badge

เปลี่ยนจาก v0.1: เดิม alert banner อยู่บนสุดเหนือ search hero — รอบนี้ search hero เป็น hero บนสุด แล้ว alert banner อยู่ใต้ลงมา

2 · Search Hero + Real-Time Metrics

Hero card บนสุด (dark navy gradient, text ขาว). Desktop = 2 คอลัมน์: ซ้าย ≈ 60% search, ขวา ≈ 40% metrics panel. Mobile = stack (metrics อยู่ใต้ search).

2a · Left — Search

  • Badge บนสุด: "LIVE DISASTER COORDINATION LINK" + activity/pulse icon (สีเขียว) — แสดงสถานะระบบ live
  • Heading (2 บรรทัด): "ระบบข้อมูลผู้ประสบภัยและสืบค้นสถานะเข้าพักฉุกเฉิน" — บรรทัด 2 highlight สีฟ้าอ่อน
  • Description: "เชื่อมต่อ อำนวยความสะดวก และรายงานความช่วยเหลือสาธารณะเพื่อความโปร่งใสแบบตามเวลาจริง (Real-Time Transparency) สามารถสืบหาญาติ ติดตามผู้ประสบภัย หรือร่วมแจกจ่ายเสบียงผ่านเครือข่ายของเรา"
  • Search input: placeholder "สืบหาญาติด้วย ชื่อ-นามสกุล หรือ รหัสประชาชน..." — full width บน mobile
  • CTA button: "ค้นหาญาติตอนนี้" (สีฟ้า/primary) เปลี่ยนจาก "ค้นหา" (v0.1)

Search Behavior

  • Submit (button click หรือกด Enter) → navigate ไป /search?q={encoded_query}
  • Validation ฝั่ง client: ต้องมีข้อความ ≥ 2 ตัวอักษรก่อน submit (ไม่แสดง error จนกว่าจะ submit)
  • ไม่ทำ live search / autocomplete บนหน้านี้ — ทำบน /search เท่านั้น
  • ข้อมูลที่พิมพ์ไม่ส่งไปไหนจนกว่า user จะ submit (privacy — ไม่ log keystroke)
Logic การค้นหาและ privacy rule (นามสกุล masking, rate-limit, search_audit) อยู่ใน FAM section ของ public-tier-flow-spec

2b · Right — Real-Time Metrics Panel ใหม่ใน v0.2

Panel แสดงตัวเลขสถานการณ์รวมแบบ aggregate. Header: "สถานการณ์ปัจจุบัน ณ ขณะนี้ (REAL-TIME METRICS)"

Metric label (TH) Value format API field Notes
ศูนย์อพยพหลัก {open} / {total} แห่ง (เช่น 5 / 5) shelters_open / shelters_total จำนวนศูนย์ที่เปิด เทียบทั้งหมด
ผู้ประสบภัยปลอดภัย {count} คน (เช่น 7) occupancy_total = occupancy รวมทุกศูนย์ (จำนวนคนที่เข้าพักรวม) — aggregate เท่านั้น. APPROVED OP-6 แสดงได้ (เพื่อให้รู้ว่าศูนย์เต็มหรือยัง); ห้าม drill-down รายชื่อ
อาสาสมัครลงปฏิบัติงาน {count} นาย + badge "ACTIVE DUTY" volunteers_active DISABLED v0.3 ซ่อนไว้ก่อนพร้อม volunteer feature — เปิดภายหลัง
v0.3: metric "อาสาสมัครลงปฏิบัติงาน" ซ่อนไว้ตอนนี้ (volunteer disabled). Panel เหลือ 2 metric: ศูนย์อพยพหลัก + ผู้ประสบภัยปลอดภัย

Metrics Behavior

  • Data source: GET /public/v1/transparency/summary — ส่ง aggregate counts (central-only)
  • Refresh: polling ทุก 10 นาที (OP-7) — ไม่มี realtime socket; public metrics ยอมรับ polling ได้
  • Stale-data threshold = 30 นาที (OP-7): ถ้า as_of เก่ากว่า 30 นาที (poll ล้มเหลวต่อเนื่อง) ให้แสดง badge/label "ข้อมูลอาจไม่เป็นปัจจุบัน" คู่กับตัวเลขเดิม — ยังโชว์ค่าเก่าได้แต่เตือน user
  • ต้องแนบ as_of timestamp จาก API — UI แสดง "อัปเดตล่าสุด …" ใต้ panel
  • Loading: แสดง skeleton ต่อ metric; ไม่บล็อก paint ของ search
  • Fail/offline (ไม่มีค่าเดิมเลย): แสดง "—" แทนตัวเลข + tooltip "ข้อมูลสดไม่พร้อมใช้งาน" (ไม่ throw error ทั้งหน้า)
Privacy note (APPROVED OP-6 — ดู CR-005 §A): "ผู้ประสบภัยปลอดภัย" = occupancy รวมทุกศูนย์ — แสดงต่อ public ได้ (aggregate ระดับระบบ; ห้าม drill-down เป็นรายชื่อ). feature flag public_metrics_occupancy (db config) เหลือไว้เป็น kill-switch, default on. metric shelters_open/total เปิดได้ปกติ

3 · Emergency Alert Banner

อยู่ ใต้ search hero (เปลี่ยนจาก v0.1 ที่อยู่บนสุด). แสดงเฉพาะเมื่อ API ส่ง active alert — ถ้าไม่มี alert ให้ซ่อน section นี้ทั้งหมด

Data Source

อ่านจาก GET /public/v1/alerts/active — ส่งคืน active alert (level, title, description, affected_area, shelter_statuses[])

Visual Design

  • Background: สีแดงอ่อน/pink (level ≥ 4) หรือสีเหลือง/ส้ม (level 1–3); มี border สีตาม level
  • Icon: triangle warning (สีแดง level 4, ส้ม level 2–3, เหลือง level 1)
  • Title: dot indicator + "{title} ระดับ {level} ({label})" — ตัวอย่างจาก mockup: "ศูนย์ประสานแจ้งพิกัดและประมวลความคลาดเคลื่อนระดับ 4 (อพยพทันที)"
  • Description: ข้อความบรรยายพื้นที่ผลกระทบ + คำสั่ง (multiline) — มาจาก field description

Alert Level Label Mapping

level label (TH) Banner color
1 เฝ้าระวัง yellow / warning
2 แจ้งเตือน orange
3 เตรียมพร้อม orange-red
4 อพยพทันที red / error

Shelter Capacity Chips

แสดง chip per shelter ใน shelter_statuses[] จาก API (ไม่ hardcode). แต่ละ chip มี dot indicator + ชื่อ + capacity badge:

Field Display
name dot + ชื่อศูนย์ + ป้าย (Primary / Secondary / …) — ตัวอย่าง "ศูนย์ค่ายทหาร (Primary)"
occupancy_pct badge สีแดงถ้า ≥ 90%, ส้ม 70–89%, เขียว <70%
label ข้อความจาก API เช่น "หนาแน่นสูง (95%)", "เปิดรับได้ (40%)"

Interaction

  • คลิกที่ chip → navigate ไป /shelters/{shelter_id}
  • Banner ไม่มีปุ่ม dismiss — ต้องเห็นตลอดที่ alert active
  • Refresh ข้อมูล alert ทุก 5 นาที (polling)
  • Mobile + alert level 4 (OP-4): banner sticky top — pin ไว้บนสุดของ viewport เมื่อ scroll (fixed) เพื่อให้คำสั่งอพยพอยู่ในสายตาตลอด. Level 1–3 scroll ตามปกติ

Card Design Rules

  • Card มี white background, rounded, subtle shadow
  • Icon สีต่างกันต่อ card (shield=red, gift=blue, person+=green, search=purple) เพื่อ scan ได้เร็ว
  • Badge อยู่ inline ข้าง heading; สีตามความหมาย (ด่วนที่สุด=red, Wishlist=blue, ร่วมแรงกาย=green, PDPA Shield=purple)
  • "PDPA Shield" badge สื่อว่า flow ค้นหาญาติคุ้มครองข้อมูลส่วนบุคคลตาม PDPA — ไม่ใช่ปุ่ม, เป็น trust marker เฉย ๆ
  • Action link style: text link พร้อม > chevron; card 2 มี primary button (CTA หลัก) + link รอง
  • ถ้า route ยังไม่พร้อม ให้แสดง link เป็น disabled (ไม่ซ่อน) พร้อม badge "เร็ว ๆ นี้"

5a · Help Center (EOC Help Center & FAQ)

Panel ซ้าย (desktop 50%). Heading: "ศูนย์รวมความช่วยเหลือ (EOC Help Center & FAQ)"

Accessibility & Performance Requirements

  • WCAG 2.1 AA — contrast ratio ≥ 4.5:1 ทุก text; hotline buttons และ metrics บน dark bg ≥ 4.5:1
  • Search input ต้องมี aria-label และ associate กับ label text
  • Alert banner ต้อง announce ผ่าน role="alert" เมื่อ content เปลี่ยน
  • Real-time metrics ที่อัปเดตเอง ใช้ aria-live="polite" เพื่อไม่รบกวน screen reader
  • Quick service cards ต้องใช้ <a> ไม่ใช่ <div onclick>
  • LCP target: ≤ 2.5 s บน 3G (static shell ต้อง hydrate เร็ว; metrics/alert API ต้องไม่บล็อก paint)
  • ภาษาไทย: lang="th" บน root element เมื่อ locale = TH

Flow Rules

  • Unsubmitted drafts stay local-only and must never replicate
  • Submitted offline records are written to local PouchDB first and must carry deterministic `client_submission_id`, `offline_evacuee_code`, shelter code, device metadata, masked sensitive payload and bounded document size
  • When Central is online, active remote is Central only; when Central/WAN is unavailable and LAN Edge is reachable, active remote switches to Edge only; when neither is reachable, active remote is none/local-only
  • Do not run long-lived replication from one device to Central and Edge simultaneously; switching remote must stop the old replication before starting the new one
  • Central and Edge `_session` cookies are separate; Edge fallback session does not authorize Central service API calls
  • Edge keeps fallback replicas and syncs backlog to Central when WAN returns; worker import reads canonical backlog from Central after failback
  • Worker owns import into MongoDB and official ID issuance; direct client write to MongoDB is forbidden in offline path
  • `sync_ack` must carry official ID mapping and cleanup requirement; UI must not silently keep sensitive local fields after import
  • `needs_review` must be actionable through a defined review path, not a dead-end status badge
  • Manual/Excel fallback exists in parallel for device failure, severe outage, or operational override

Assisted Import Flow

  • User captures data in fixed manual form or Excel template aligned to MVP field contract
  • Authorized operator uploads or enters batch for `import_batches`
  • System maps template version and validates required fields
  • Rows with issues enter review before confirm; import is not blind bulk insert
  • Confirmed rows create/update `evacuees` and preserve `source=excel_import` or `manual_entry` metadata for audit

API and Data Intent

  • Data/doc types: `draft_registration`, `offline_submission`, `sync_ack`, `sync_review`, `cleanup_confirmation`, `offline_submissions`, `import_batches`
  • PWA queue UI needs local state merged with worker-returned review/ack docs and the current `active_remote` provenance
  • Review surface likely needs `GET /offline-submissions` and resolve/resubmit affordances in API contract
  • Fallback import needs `POST /imports/fallback` and batch/status endpoints or equivalent workflow
  • Every path must preserve audit metadata, shelter scope, `client_submission_id`, and remote provenance (`central`, `edge_fallback`, `local_only`)

Acceptance Criteria

  • Central online: registration/screening writes local PouchDB first, syncs to Central as the only active remote, and does not start Edge replication
  • Central unavailable + LAN fallback: client switches to Edge as the only active remote, queues/syncs submitted records there, and Central API calls remain blocked until failback
  • No remote/local-only: client continues allowed draft/queue work locally, marks sync paused, and resumes with Central preferred when any remote returns
  • Remote switching stops old replication before starting the new target and preserves checkpoints/client IDs so the same offline submission never creates duplicate official records
  • Imported offline record returns official ID mapping and updates local queue state
  • Duplicate or invalid offline record surfaces explicit review/rejection state
  • Manual/Excel fallback can be mapped and reviewed before source-of-truth import

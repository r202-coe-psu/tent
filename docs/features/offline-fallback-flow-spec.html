Remote Mode State Machine

Mode Entry Condition Active Endpoint Exit Condition
`central_online` Central reachable and Central session valid Central CouchDB only Central unreachable or Central session invalid
`edge_fallback` Central unavailable and LAN Edge reachable with Edge session Edge CouchDB only Central recovers or Edge unavailable
`disconnected` Neither Central nor Edge reachable none (no write endpoint) Central or Edge becomes reachable with valid session
`failback_pending` Central recovers while client was using Edge transition Edge -> Central Edge flow stopped, Central session validated, Central flow resumed

Submission State Machine

State Owner Entry Condition Exit Condition
`editing` PWA UI user is filling registration/screening payload submit or cancel
`sending` PWA + active endpoint client posts typed `offline_submission` with deterministic `client_submission_id` to active endpoint accepted by Central/Edge or send fails
`retry_waiting` PWA + sync manager send failed because endpoint/session/network unavailable automatic retry (max 3 attempts) after reconnect/session restore, then show cannot-connect banner; user can force retry
`received` Central or Edge CouchDB + worker trace `offline_submission` accepted on active endpoint; Edge backlog forwards to Central after failback worker consumes and stages from Central
`validating` sync worker staged submission under schema/scope/duplicate checks imported, needs_review, or rejected
`imported` worker + MongoDB official record written and ID mapped PWA receives `sync_ack` and sends cleanup confirmation
`needs_review` worker + review actor ambiguous duplicate or permission/data issue that can be resolved resolved resubmission or supervised import decision
`rejected` worker invalid doc or unresolvable import failure new corrected submission only

Flow Rules

  • Unsubmitted form data is UI state only and must not be treated as a durable write queue
  • Submitted records must be sent to the active endpoint immediately and carry deterministic `client_submission_id`, `offline_evacuee_code`, shelter code, device metadata, masked sensitive payload, and bounded document size
  • When Central is online, active endpoint is Central only; when Central is unavailable and LAN Edge is reachable, active endpoint switches to Edge only; when neither is reachable, active endpoint is none and writes remain pending retry
  • Do not run active write/sync flow to Central and Edge simultaneously; failover/failback must stop the old flow before starting the new one
  • Central and Edge `_session` cookies are separate; Edge fallback session does not authorize Central service API calls
  • Edge keeps fallback replicas and syncs backlog to Central when WAN returns; worker import reads canonical backlog from Central after failback
  • When no endpoint is reachable, submission attempts must fail with explicit disconnected status-only mode (no read-only local cache), run automatic retry up to 3 attempts, then show a clear cannot-connect banner with force retry action
  • Worker owns import into MongoDB and official ID issuance; direct client write to MongoDB is forbidden in this path
  • `sync_ack` must carry official ID mapping and cleanup requirement; UI must not silently keep sensitive local fields after import
  • `needs_review` must be actionable through a defined review path, not a dead-end status badge
  • Manual/Excel fallback exists in parallel for device failure, severe outage, or operational override

Assisted Import Flow

  • User captures data in fixed manual form or Excel template aligned to MVP field contract
  • Authorized operator uploads or enters batch for `import_batches`
  • System maps template version and validates required fields
  • Rows with issues enter review before confirm; import is not blind bulk insert
  • Confirmed rows create/update `evacuees` and preserve `source=excel_import` or `manual_entry` metadata for audit

API and Data Intent

  • Data/doc types: `offline_submission`, `sync_ack`, `sync_review`, `cleanup_confirmation`, `offline_submissions`, `import_batches`
  • Submission status UI needs send/retry/bannering state merged with worker-returned review/ack docs, current `active_endpoint` provenance, and app-level event channel updates
  • Review surface likely needs `GET /offline-submissions` and resolve/resubmit affordances in API contract
  • Fallback import needs `POST /imports/fallback` and batch/status endpoints or equivalent workflow
  • Every path must preserve audit metadata, shelter scope, `client_submission_id`, and endpoint provenance (`central`, `edge_fallback`, `disconnected`)

Acceptance Criteria

  • Central online: registration/screening submits to Central as the only active endpoint and does not start Edge write flow
  • Central unavailable + LAN fallback: client fails over to Edge as the only active endpoint, submits there, and Central API calls remain blocked until failback
  • Neither endpoint reachable: submission attempt returns explicit disconnected status-only mode (no read-only local cache); client automatically retries up to 3 attempts, then shows clear cannot-connect banner and allows force retry
  • Failover/failback stops old flow before starting the new target and preserves `client_submission_id` so the same submission never creates duplicate official records
  • Imported offline record returns official ID mapping and updates submission status
  • Duplicate or invalid offline record surfaces explicit review/rejection state
  • Manual/Excel fallback can be mapped and reviewed before source-of-truth import

Public Portal — Donation & Queue Booking (บริจาคและจองคิว)

Back Landing Spec Public Tier Flow Draft v0.1 2026-06-22

Summary

หน้า "บริจาคและจองคิว" — nav link ของ Public Portal (landing hand off มา ที่ /donate). เป็นจุดที่ ผู้บริจาค (Donor) ดูว่าศูนย์ใดขาดอะไร, ประกาศของที่จะนำมามอบ, และ จองคิว/นัดเวลาส่งของล่วงหน้า เพื่อให้ทีมงานที่ศูนย์ เตรียมจุดรับและลดความแออัดหน้าศูนย์. นี่คือ DN tier (Donor) ของ public/no-login service (ดู Public Tier Flow Spec §DN, FD-16).

หน้านี้เป็น Public / No-Login Tier (DN) — บริจาคได้โดยไม่ต้อง login. donor ให้ ชื่อ + เบอร์โทร เพื่อสร้าง pre-declaration + soft reservation; ระบบตอบ ตั๋ว (booking ref + QR + tracking token). อ่าน/เขียน central-only — trust boundary ดู Public Tier Flow Spec
เป้าหมายหลัก: เติมเต็มของที่ศูนย์ขาดให้ตรงจุด + ติดตามของได้ ทั้งสองฝั่ง — donor พิมพ์ตั๋วแปะลงของ, ศูนย์เห็นรายการรอรับ พร้อมเวลานัด/เวลาคาดว่าจะถึง. ออกแบบให้ใช้บนมือถือใน network ไม่เสถียร; รองรับ TH/EN (EN toggle)
Change tracking: หน้านี้และ field/exposure ใหม่ของมัน tracked ใน CR-005 §F. mockup เพิ่ม donor PII (Line/email/ใบอนุโมทนา), logistics (วิธีส่ง/ยานพาหนะ), และ slot-based queue booking เกินร่างเดิม ใน public-tier-flow-spec §DN — รายละเอียดที่ยังต้องเคาะดูส่วน Open Points

Layout Overview — 4-Step Wizard

หน้านี้เป็น stepper 4 ขั้น (tab bar ด้านบนแสดงสถานะแต่ละขั้น). ทิศทาง ปกติเดินจากซ้ายไปขวา; กดย้อนขั้นที่ทำเสร็จแล้วได้.

  1. ความต้องการด่วน (Urgent Needs Board) — ดูของที่ ทุกศูนย์ขาด, filter/ค้นหา, เลือก "จองคิวบริจาค" จากการ์ดศูนย์
  2. ฟอร์มบริจาค (Donation Form) — ข้อมูลผู้บริจาค + รายการสิ่งของ (pre-fill จากการ์ดที่เลือกในขั้น 1)
  3. เวลา/สถานที่ (Logistics) — วิธีจัดส่ง, ยานพาหนะ, ศูนย์ปลายทาง (lock จากการ์ด), เลือกวัน + ช่วงเวลา (จองคิว) + เวลา คาดว่าจะถึง
  4. ตั๋วของฉัน (My Ticket) — QR + booking ref + สรุปนัด; บันทึก/พิมพ์ตั๋วแปะลงของ

1 · ความต้องการด่วน — Urgent Needs Board

ตอบโจทย์ข้อ 1: เห็นรายการที่ศูนย์ต้องการรับบริจาคทั้งหมด. อ่าน aggregate จาก GET /public/v1/needs (no auth) — ไม่มี ข้อมูลรายบุคคล.

  • หัวข้อ "กระดานความต้องการด่วน" + subtitle "อัปเดตข้อมูลแบบเรียลไทม์จากทุกศูนย์พักพิง คุณสามารถช่วยเติมเต็มในส่วน ที่ขาดแคลนได้ทันที"
  • ช่องค้นหา สิ่งของ (placeholder "ค้นหาสิ่งของที่ต้องการ บริจาค เช่น น้ำดื่ม, ยาพาหนะ…") — filter client-side ตามชื่อ item
  • Category filter chips: ทั้งหมด · ● ด่วนพิเศษ · อาหาร & น้ำ · ยารักษาโรค (ชุด category มาจาก needs data)
  • การ์ดต่อศูนย์ — ชื่อศูนย์ + badge "เปิดรับบริจาค", และรายการ need item แต่ละรายการพร้อมระดับความเร่งด่วนและจำนวนที่ขาด:
    • item ด่วน (พื้นแดง, ● จุดแดง): เช่น "ด่วน! น้ำดื่ม — ขาด 500 แพ็ค", "ด่วน! ยาแก้ปวด — ขาด 200 แผง"
    • item ปกติ: เช่น "ข้าวกล่อง — ขาด 300 ชุด"
    • item ล้นสต็อก / งดรับ (เทาจาง): เช่น "งดรับเสื้อผ้ามือสอง — ล้นสต็อก 120%" — แสดงเพื่อกันบริจาคซ้ำ, กดเลือกไม่ได้ (DN-4)
  • ปุ่ม "จองคิวบริจาค →" ต่อการ์ด + caption "ช่วยลดความแออัดหน้าศูนย์" → ไปขั้น 2 โดย lock ศูนย์ปลายทาง และ pre-fill รายการสิ่งของจาก need ที่กด (deep-link ?shelter=/?need=)
ความเร่งด่วน + จำนวนที่ขาด + สถานะ "งดรับ/ล้นสต็อก" เป็น aggregate ต่อ need item — ไม่มีข้อมูลรายบุคคล. field/สูตรคำนวณ "ขาด N" / "ล้น %" รอ Data Model (DN-4)

2 · ฟอร์มบริจาค — Donation Form

ตอบโจทย์ข้อ 2: กรอกสิ่งที่จะบริจาค จากการ์ดที่เลือก. หมวดหมู่/ประเภทสิ่งของ pre-fill จาก need item ที่กดมา (donor แก้ไข/เพิ่ม รายการได้).

ส่วนที่ 1 — ข้อมูลผู้บริจาค ("สำหรับติดต่อกลับกรณีฉุกเฉิน")

Field Required หมายเหตุ
ชื่อ-นามสกุล / นามแฝง / องค์กร required รับนามแฝง/ชื่อองค์กรได้ (ไม่บังคับชื่อจริง)
เบอร์โทรศัพท์มือถือ required สำหรับส่ง SMS ยืนยัน; index ด้วย hash ของเบอร์ — ไม่ echo เบอร์กลับใน response
Line ID optional ช่องทางติดต่อเสริม — เก็บได้ (optional), ยืนยัน DN-2 (2026-06-22)
อีเมล optional ช่องทางติดต่อเสริม — เก็บได้ (optional), ยืนยัน DN-2 (2026-06-22)
☑ ต้องการใบอนุโมทนาบัตร / ลดหย่อนภาษี checkbox ถ้าติ๊ก → ต้องเก็บ PII เพิ่ม (ชื่อจริง/ที่อยู่/เลขผู้เสียภาษี) + flow ออกใบ — ยังไม่เคาะ (DN-3)

ส่วนที่ 2 — รายละเอียดสิ่งของบริจาค (repeatable)

แต่ละแถวรายการของบริจาค (กด "เพิ่มรายการสิ่งของ" เพื่อเพิ่มแถว):

  • หมวดหมู่ (category) — เช่น Food, Medicine (pre-fill จาก need)
  • ประเภทสิ่งของ (type) — เช่น "น้ำดื่มขวด 600ml" (pre-fill จาก need)
  • ปริมาณ (quantity) — ตัวเลข
  • หน่วย (unit) — เช่น ชิ้น / แพ็ค / แผง / ชุด
  • สภาพสิ่งของ (condition) — เช่น "ของใหม่ 100%"
  • หมายเหตุเพิ่มเติม (notes, optional) — เช่น "ข้าวกล่องมังสวิรัติ"
  • ปุ่ม "+ เพิ่มรายการสิ่งของ" — เพิ่มแถว item; ลบแถวได้ (เหลืออย่างน้อย 1)
  • ปุ่ม "ถัดไป: เลือกจุดส่งมอบ →" — validate (ชื่อ/เบอร์/≥1 item) แล้วไปขั้น 3

3 · เวลา/สถานที่ — Logistics & Queue Booking

ตอบโจทย์ข้อ 3: ระบุเวลาที่จะส่ง + ใครเป็นคนส่ง + ส่งไปที่ไหน (lock จากการ์ด) + เวลาที่คาดว่าของจะถึง เพื่อให้ทีมงานที่ศูนย์ ติดตามได้.

  • วิธีการจัดส่ง * — เลือก 1:
    • นำมาส่งด้วยตนเอง
    • ส่งผ่านขนส่งพัสดุ
    • ต้องการให้รถศูนย์ไปรับ (ของเยอะมาก)
  • ประเภทยานพาหนะที่จะนำมาส่ง * (caption: "ข้อมูลนี้ สำคัญมาก เพื่อให้จุดรับของกะพื้นที่จอดและเตรียมคนยกของ"): รถจักรยานยนต์ / รถเก๋ง-รถยนต์ / รถกระบะ / รถบรรทุก — แสดงเมื่อวิธีส่ง = ตนเอง/รถศูนย์
  • เลือกศูนย์รับบริจาค * — lock จากการ์ดที่กดมาในขั้น 1 (read-only) เช่น "ศูนย์พักพิง เทศบาลนครหาดใหญ่ (โรงเรียนเทศบาล 2) (ของยังรับได้ 60%)". "ของยังรับได้ %" = donation receiving capacity (DN-4)
  • เลือกวันที่และช่วงเวลาส่งของ * — grid slot ต่อช่วง เช่น 09:00-10:00 / 10:00-11:00 / 13:00-14:00 / 14:00-15:00 / 15:00-16:00. แต่ละ slot แสดงสถานะ "ว่าง" หรือ "คิวเต็ม (งด)" (disabled). slot ที่เลือก = นัดหมาย (จองคิว). ศูนย์เป็นผู้ตั้งค่า slot เอง — ช่วงเวลา/ จำนวน slot/ความจุต่อ slot กำหนดโดยเจ้าหน้าที่ศูนย์ใน Donation module (back-office); หน้า public อ่าน slot + ความจุที่เหลือ แล้ว disable เมื่อเต็ม (DN-5, ยืนยัน 2026-06-22)

"ใครเป็นคนส่ง" + "เวลาคาดว่าจะถึง" (ข้อ 3)

  • ผู้ส่ง = ชื่อ + เบอร์ผู้บริจาคจากขั้น 2 (ผูกกับ declaration เพื่อให้ศูนย์รู้ว่าใครจะมา). ผู้ส่งกรอกข้อมูล ทั้งหมดเอง (DN-6, ยืนยัน 2026-06-22)
  • ถ้าวิธีส่ง = ขนส่งพัสดุ → เลข tracking ของผู้ให้ บริการมักได้มาทีหลัง donor จึง ยังไม่ต้องกรอกตอนนี้; donor กลับมา กรอก/แก้เลข tracking ภายหลังผ่านหน้า ticket (ลิงก์ /donate/track/{tracking_token} — ดูขั้น 4)
  • เวลาคาดว่าของจะถึง (ETA) = ช่วงเวลา slot ที่จอง (นัดหมาย). ศูนย์ใช้ ETA นี้เตรียมจุดรับและติดตามว่าของถึงตามนัดหรือไม่. field eta/slot ติดไปกับ declaration doc
  • ปุ่ม "กลับ" (ขั้น 2) · "ยืนยันการจองคิวบริจาค" → ยิง POST /public/v1/donations, ได้ตั๋ว (ขั้น 4)

4 · ตั๋วของฉัน — My Ticket

ตอบโจทย์ข้อ 4: ผู้บริจาคบันทึก + พิมพ์ตั๋วแปะลงของที่บริจาค เพื่อให้ติดตามได้ทั้งฝั่งศูนย์และผู้บริจาค.

  • การ์ดสำเร็จ (พื้นเขียว, icon โล่): "สำเร็จ! ตั๋วบริจาคของคุณ" + คำแนะนำ "แสดง QR Code นี้ให้ รปภ. หรือเจ้าหน้าที่ศูนย์"
  • QR Code — encode ลิงก์ติดตาม /donate/track/{tracking_token} (เปิด GET donations endpoint); สแกนได้ทั้งฝั่งศูนย์ (ยืนยันรับของ) และ donor (ดูสถานะ)
  • BOOKING REF. เช่น DN-306892 — รหัส อ่านได้/พิมพ์ได้ (มนุษย์อ่าน) คู่กับ tracking_token (opaque)
  • สรุปนัด: ศูนย์พักพิงปลายทาง (ชื่อศูนย์), เวลานัดหมาย (ช่วง slot เช่น 09:00-10:00), รายการสิ่งของ (เช่น "1 รายการ")
  • ปุ่ม "นำทางด้วย Google Maps" (เปิด external map ไปศูนย์)
  • "บันทึก/พิมพ์ตั๋ว" (ข้อ 4) — ปุ่มพิมพ์/บันทึกตั๋ว (print-friendly label: QR + booking ref + ศูนย์ + เวลานัด + รายการ) เพื่อ แปะลงกล่อง/ถุงของบริจาค. donor แนะนำให้เก็บลิงก์ tracking ไว้ดูสถานะ (declared → received / expired)
  • ลิงก์ "กลับสู่หน้าหลัก"

หน้า Ticket เปิด/แก้ไขภายหลัง — /donate/track/{tracking_token} (DN-6)

ตั๋ว (QR/ลิงก์) เปิดซ้ำได้ตลอดด้วย tracking_token และ แก้ไขบางฟิลด์ได้ เพราะ donor มักได้ข้อมูลบางส่วนทีหลัง:

  • เลข tracking ขนส่งพัสดุ — กรอก/แก้ได้ภายหลัง (กรณี วิธีส่ง = ขนส่งพัสดุ) → ศูนย์เห็นเลขเพื่อติดตามพัสดุ
  • ดูสถานะ declaration ของตน: declared → received / expired
  • แก้ได้เฉพาะก่อนศูนย์ mark received; ทุกการแก้เขียน central → replicate ลง shelter db (ศูนย์เห็น update)
  • ยิง PATCH /public/v1/donations/{tracking_token} (auth = token ในตัว); แก้ได้เฉพาะ field ที่อนุญาต (เลข tracking / หมายเหตุ), ห้ามแก้ ศูนย์ปลายทาง/slot/ตัว booking_ref
ติดตามสองฝั่ง: declaration เขียนที่ central → replicate ลง shelter db ปลายทาง — ศูนย์เห็นรายการรอรับ + เวลานัด/ETA ใน Donation module; สแกน QR บนของเพื่อ mark received. donor เปิดลิงก์ tracking_token ดูสถานะของตนเอง (เห็นเฉพาะของตน)

5 · Data Exposure & Privacy (DN tier)

mockup ขยาย field ของ DN tier เกินร่างเดิมใน public-tier-flow-spec §DN (เดิมเก็บ donor PII ขั้นต่ำ = ชื่อ + เบอร์). mockup เพิ่ม Line/email, ใบอนุโมทนา (ลดหย่อนภาษี), logistics และ slot booking — tracked ใน CR-005 §F

GET donations/{token} สถานะ declaration ของตนเท่านั้น OK เห็นเฉพาะของตน (declared / received / expired)

หลักการ (เหมือน DN tier)

  • เก็บ donor PII ขั้นต่ำ; index เบอร์ด้วย hash — ไม่ echo เบอร์กลับ
  • submit ผ่าน POST (PII ไม่ติด URL/access log)
  • tracking_token = opaque random ≥128-bit, ใช้ได้ตลอดไป — แสดง/แนะนำให้เก็บ
  • response ไม่มี field ภายใน (_id/_rev/doc type/internal id)
  • เขียน central → replicate ลง shelter db ปลายทาง; rate-limit + audit; ไม่ fallback edge

Donation doc shape (DN-7, ยืนยัน 2026-06-22)

Field-level canonical = docs/data/schema.md §2.3 donation (schema_v 2) + §2.13 donation_slot (slot ที่ศูนย์ตั้งค่า, DN-5). doc type:"donation" เดิมถูก extend (ไม่ใช่ doc ใหม่) — เพิ่ม logistics, booking_ref, donor.line_id/email, items[].category/condition/note, tax_receipt_requested. เขียนที่ shelter db (public flow ผ่าน central → replicate). response สาธารณะ project เฉพาะ field ที่อนุญาต (ไม่ส่ง _id/_rev/phone/phone_hash/tracking_token_hash):

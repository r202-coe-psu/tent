Flow Rules

  • Dashboard Landing is the first screen for shelter_manager and viewer-like roles after shell load when product chooses dashboard-first navigation
  • Summary cards must at least cover current occupancy, capacity percent or over-capacity count, vulnerable count, fast-track count, in/out today, medical summary aggregate, `last_updated`
  • UI must present stale awareness using `last_updated`; it must not pretend the dashboard is fully real-time when read model is slightly behind
  • Export Request Modal must reflect role scope and masking policy before submit
  • Export Status View must show auditable metadata even if Sprint 0 or deployment phase still limits file generation

API and Data Intent

  • Dashboard API: `GET /dashboard/summary` with shelter/date filters as allowed
  • Export API: `POST /exports`, `GET /exports/{id}`
  • Primary data sources: `dashboard_summaries`, `movement_events`, `evacuees`, `export_jobs`, `audit_logs`
  • Export payload/response must persist `report_type`, `format`, `filters_snapshot`, `masking_policy_applied`, `status`, and optional `file_ref`
  • Dashboard response should be explicitly versioned enough for frontend cards and QA assertions

Acceptance Criteria

  • Dashboard occupancy and in/out counts reconcile with movement events in UAT data
  • Dashboard always shows `last_updated`
  • Unauthorized role cannot obtain medical detail or unmasked export by UI or direct API call
  • Export request writes audit evidence including requester, shelter scope and masking policy
  • If file generation is disabled or pending by environment phase, export metadata/status still works without implying file readiness falsely

Open Questions

  • Concurrent-user and latency targets still need numeric thresholds
  • Environment-specific decision is still needed on whether July export gate requires actual file generation or metadata/trace only

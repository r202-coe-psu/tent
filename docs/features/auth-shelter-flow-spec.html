Screen Flow

  • `Login -> Central session -> Shelter Context Gate -> Home Shell` คือ happy path ของทุก role
  • ถ้า Central ใช้ไม่ได้แต่ LAN Edge reachable ให้สร้าง Edge fallback session แยก และ active remote ของ PouchDB เปลี่ยนเป็น Edge เท่านั้น
  • ถ้า Central และ Edge ใช้ไม่ได้ ให้เข้า local-only เฉพาะงานที่มี local capability; protected Central API action ต้อง block ด้วยข้อความควบคุมได้
  • เมื่อ Central กลับมา ให้ client หยุด Edge replication, re-check Central session/refresh login ถ้าจำเป็น, switch active remote กลับ Central และให้ Edge sync backlog ขึ้น Central
  • ถ้า user มี assigned shelter เดียว ให้ระบบ set active shelter อัตโนมัติและ redirect ทันที
  • ถ้า user ไม่มี assigned shelter และไม่ใช่ system admin ให้ deny ทั้ง UI และ API
  • ทุก protected page ต้องเช็ค expired session ก่อน render feature data
  • หน้า Shelter Master มีเฉพาะ admin/system role ที่ได้รับสิทธิ์; registration/medical/viewer เห็นได้แค่ shelter metadata ที่จำเป็นต่อ workflow

API and Data Intent

  • Auth endpoints: `POST /auth/login`, `POST /auth/logout`, `GET /auth/me`; Central is attempted before any Edge fallback endpoint
  • Central and Edge CouchDB `_session` cookies are separate browser credentials; Edge fallback cookie must never be accepted by Central service API/proxy
  • Client sync manager stores `active_remote={central|edge|none}` and never keeps long-lived replication to Central and Edge at the same time
  • Shelter context must return actor identity, `role_keys`, `assigned_shelter_ids`, current shelter metadata, and remote provenance (`central` or `edge_fallback`) needed by the shell
  • Shelter master endpoints must own `shelters.status`, `capacity.total`, `capacity.soft_limit`, `allow_over_capacity`
  • 403 path must return stable permission failure shape so UI can render blocked state instead of generic error
  • Audit must capture login success/failure, fallback entry, failback, shelter status changes, and override reasons where policy requires

Acceptance Criteria

  • Central success: with Central reachable and valid credentials, user gets Central session, active remote is Central only, shelter scope loads, and no Edge replication remains running
  • Central down + Edge success: with WAN/Central unavailable and LAN Edge reachable, user gets Edge fallback session, active remote is Edge only, Central API calls stay blocked, and queued local writes sync to Edge
  • Failback: when Central recovers, client stops Edge replication before starting Central replication, validates/refreshes Central session, Edge syncs backlog to Central, and duplicate official records are not created
  • User with wrong credentials stays on Login and sees explicit failure state
  • User with one assigned shelter lands in the correct scope without manual chooser
  • User cannot read or mutate data outside assigned shelter by direct API call
  • Closed shelter blocks normal new check-in unless authorized override path is used with reason
  • Session expiry returns user to Login without exposing stale protected content

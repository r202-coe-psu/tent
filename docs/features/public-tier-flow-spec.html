Feature Flow Spec

Public / No-Login Tier (FAM + DN + PUB)

BackFamily Search SpecDonation SpecDraft v0.32026-06-22

Summary

ครอบคลุม access tier ที่ไม่ต้อง login: Family Search (FAM), Donor (DN, FD-16) และ Public Transparency (PUB) — เป็น Public API Service แยกจากระบบหลัก รันที่ central เท่านั้น

หลักการ: เห็นน้อยที่สุดเท่าที่จำเป็น · อ่านจาก central node เสมอ · ไม่ expose CouchDB ตรงสู่ public · ไม่แตะ edge/shelter device · staff LAN fallback ไม่ใช้กับ public tier
API tier (machine / API key / EOC / Open API) ไม่อยู่ใน spec นี้ — deferred ตาม FD-14 (ทำหลัง main application เสร็จ)

Architecture & Trust Boundary

  • Public API Service เป็น service แยก deploy ที่ central เท่านั้น — อ่านจาก central CouchDB (replica รวมของทุก shelter db) ไม่มีเส้นทางใดไปถึง Edge หรือ shelter device
  • Public/no-login endpoints เป็น Central-only: ถ้า Central ใช้ไม่ได้ ให้ตอบ error ที่ควบคุมได้ ไม่ fallback ไป LAN Edge แม้ staff client จะอยู่ใน Edge fallback mode
  • ห้าม expose CouchDB endpoint ตรงสู่ public ทุกกรณี — ทุก request ผ่าน service layer ที่ทำ projection (เลือก field ที่อนุญาตเท่านั้น) ก่อนตอบ
  • เส้นทางเขียน (donor declaration) เขียนลง central แล้ว replicate ลง shelter db ปลายทางทิศเดียว — ไม่เขียนผ่าน Edge และไม่รอ Edge confirm
  • Data freshness = eventual ตาม replication lag — ทุก response แนบ as_of timestamp และ UI public ต้องแสดงเวลาข้อมูลล่าสุด
  • Response ของ public tier ต้องไม่มี field ภายใน (_id, _rev, doc type, internal id) — ใช้ opaque public id/token เท่านั้น

Abuse Control & Friction Policy

ช่วงภัยพิบัติ usability มาก่อน friction — ใช้ risk-based friction แทนการบังคับ OTP/CAPTCHA ทุก request:

  • ชั้นแรก (เปิดตลอด): rate-limit ต่อ IP และต่อเบอร์โทร, max result ต่อ query, reservation TTL, audit ทุก write และทุก FAM query
  • OTP: เป็น config flag public_otp_required (db config) — default off; เปิดเฉพาะเมื่อเจอ spam จริง การปิด OTP แลกกับ fake declaration ซึ่งรับได้เพราะ reservation เป็น soft + หมดอายุเอง
  • CAPTCHA: ไม่ใช้ third-party ที่พึ่ง network ภายนอก (reCAPTCHA) — ถ้าจำเป็นใช้ proof-of-work แบบ self-hosted (เช่น ALTCHA) และ trigger เฉพาะ traffic pattern ผิดปกติ ไม่โชว์กับ user ปกติ
  • Reservation TTL: declaration ที่ไม่มีการส่งของจริงภายใน 72 ชม. ถูกปลด reservation อัตโนมัติ (คืน need ให้ donor รายอื่น) — declaration ยัง track ได้แต่สถานะเป็น expired
Open point: นโยบาย "OTP default off + risk-based" เป็นข้อเสนอจากการ review (2026-06-11) แทนร่างเดิมที่บังคับ OTP/CAPTCHA — รอเคาะสุดท้าย

FAM — Family Search

Expanded surface (CR-005 §E, approved 2026-06-22): หน้า public /search ("สืบค้นญาติ") ขยาย FAM tier เกินร่างด้านล่าง — ค้นได้เพิ่มด้วย เลขบัตร 13 หลัก (exact) + passport (FS-1) และคืน field เพิ่ม เลขบัตร mask, เพศ, ชื่อศูนย์, ภูมิลำเนา mask, เวลาลงทะเบียน, โซน (FS-2, ทุก field masked) นอกเหนือ ชื่อ/สถานะด้านล่าง. กติกา mask ชื่อ (ชื่อเต็ม + สกุล mask) คงเดิม (FS-3). identifier ทุกตัว exact-only กัน enumeration. รายละเอียดเต็ม: Family Search Flow Spec
  • Endpoint: POST /public/v1/family-search — ใช้ POST กันชื่อบุคคลติด access log/URL
  • Query ได้ 2 ทาง: (a) ชื่อ/สกุล — ขั้นต่ำ 3 ตัวอักษร match ระดับ token; (b) เบอร์โทร — ต้องครบทั้งเบอร์เท่านั้น (normalize ตัดขีด/เว้นวรรค) ไม่รับ partial กัน enumeration — ไม่มี endpoint แบบ list-all
  • Response ต่อคน: ชื่อจริงเต็ม + นามสกุล masked (กติกาด้านล่าง), ชื่อเล่น, ชื่อศูนย์, สถานะ (in_shelter / moved / checked_out) — ไม่มีเบอร์โทรเด็ดขาด (แม้ query ด้วยเบอร์) ไม่มีเลขบัตร medical zone หรือ field อื่นใด
  • กติกา mask นามสกุล (นับเป็น grapheme cluster ไม่ใช่ code point — สระ/วรรณยุกต์ไทยต้องเกาะพยัญชนะ):
    • ยาว ≥ 5 ตัว: เก็บ 2 ตัวแรก + **** + 2 ตัวท้าย — "อินทปัญญา" → "อิน****ญา"
    • ยาว ≤ 4 ตัว (edge case เช่น "ลี", "แสง"): เก็บตัวแรกตัวเดียว + **** — "ลี" → "ล****"
    • **** เป็นความยาวคงที่เสมอ — ไม่เปิดเผยความยาวจริงของนามสกุล
  • Consent model = opt-out: ค้นเจอเป็น default; ถ้า privacy.search_excluded = true ให้ hide ทั้ง record จากผลค้นหา (ไม่ปรากฏแม้ query ตรงตัว)
  • Max 10 ผลต่อ query; ทุก query เขียน search_audit ที่ central (query text, IP hash, ผลลัพธ์กี่รายการ)
  • Anti-enumeration: rate-limit แรงต่อ IP, ตอบ generic เมื่อไม่พบ (ไม่บอกว่า "มีแต่ถูกซ่อน"), PoW challenge เมื่อ pattern ผิดปกติ

DN — Donor

Expanded surface (CR-005 §F, 2026-06-22): หน้า public /donate ("บริจาคและจองคิว") ขยาย DN tier เกินร่างด้านล่าง — donor PII เพิ่ม Line ID / email (optional, DN-2 resolved) + ใบอนุโมทนา (DN-3 open) นอกเหนือ ชื่อ+เบอร์; เพิ่ม logistics (วิธีส่ง/ยานพาหนะ/ETA) + slot booking ที่ ศูนย์ตั้งค่าเอง (DN-5 resolved); donor แก้เลข tracking ขนส่งพัสดุภายหลังผ่าน ticket (DN-6 resolved → PATCH /donations/{token}); doc ใหม่ type:"donation" schema_v 1 (DN-7 resolved). เหลือ DN-1 (OTP) / DN-3 / DN-4 open. รายละเอียดเต็ม: Donation & Queue Booking Flow Spec
EndpointAuthทำอะไร

POST /public/v1/donationsเบอร์โทร (+ OTP เฉพาะเมื่อ flag เปิด)สร้าง pre-declaration + soft reservation → ตอบ tracking_token

PATCH /public/v1/donations/{tracking_token}token ในตัวdonor แก้เลข tracking ขนส่งพัสดุ/หมายเหตุ ภายหลัง (DN-6); ห้ามแก้ศูนย์/slot/booking_ref; lock เมื่อ received
POST /public/v1/otp/request · /otp/verify—ใช้เฉพาะเมื่อ public_otp_required เปิด
  • tracking_token = opaque random ≥128-bit, ใช้ได้ตลอดไป (เคาะ 2026-06-11) — แสดงครั้งเดียวตอนสร้าง + แนะนำ donor เก็บลิงก์
  • เก็บ donor PII ขั้นต่ำ: ชื่อ + เบอร์โทร; index ด้วย hash ของเบอร์ ไม่ expose เบอร์กลับใน response ใด ๆ
  • declaration เขียนเป็น doc ฝั่ง central → replicate ลง shelter db ปลายทาง; ศูนย์เห็นรายการรอรับใน Donation module

PUB — Transparency

  • GET /public/v1/transparency/summary — system-wide aggregate: shelters_open/shelters_total, occupancy_total (OP-6 approved), vulnerable_count (OP-8 approved) — ดู CR-005 §A/§D; ขับ landing metrics + Shelter Dashboard cards
  • GET /public/v1/transparency/shelters — รายชื่อศูนย์ที่เปิด + occupancy รวม + capacity ต่อศูนย์ (จำนวนคน ไม่มีรายบุคคล); per-shelter แสดงตัวเลขเป๊ะเสมอ (OP-9 approved — ไม่ปัด/ไม่ mask)
  • GET /public/v1/transparency/donations — ยอดรับบริจาค aggregate ต่อศูนย์/campaign
  • Aggregate เท่านั้นทุกกรณี — ไม่มีเส้นทางใดไปถึงข้อมูลระดับบุคคลจาก tier นี้ (occupancy_total / vulnerable_count = aggregate ระดับระบบ, ห้าม drill-down)

API Conventions (service นี้)

Base path : /public/v1
Error     : { "error": { "code": "RATE_LIMITED", "message": "..." } }   // no PII in errors
Paging    : ?limit=&bookmark=        // CouchDB-style bookmark
Freshness : ทุก response แนบ "as_of": ISO-8601

Acceptance Criteria

  • ทุก endpoint ใช้งานได้โดยไม่ login; ไม่มี response ใดมี field เกินที่ระบุใน spec นี้
  • FAM: คนที่ search_excluded ไม่ปรากฏในผลค้นหาแม้ query ตรงตัว; ทุก query มี search_audit record
  • FAM: นามสกุลใน response ถูก mask ตามกติกาเสมอ (รวม edge case นามสกุลสั้น); ไม่มี response ใดมีเบอร์โทร; ค้นด้วยเบอร์ partial ต้องไม่ได้ผลลัพธ์
  • DN: declaration ที่ไม่ส่งของภายใน 72 ชม. ปลด reservation อัตโนมัติ; tracking_token เดิมยังเปิดดูสถานะ expired ได้
  • DN: เมื่อ public_otp_required ปิด — สร้าง declaration ได้โดยไม่มี OTP; เมื่อเปิด — ต้องผ่าน OTP ก่อน
  • Public service อ่าน/เขียนผ่าน central เท่านั้น — ทดสอบโดยตัด central ออก: ทุก endpoint ตอบ error ที่ควบคุมได้ ไม่ fallback ไป Edge หรือ staff LAN fallback path
  • Rate-limit ทำงานต่อ IP และต่อเบอร์; เกิน limit ตอบ RATE_LIMITED ไม่ leak ข้อมูล

Open Points

  • เคาะสุดท้าย: OTP default off + risk-based friction (ดู Abuse Control) — เสนอแทนการบังคับ OTP/CAPTCHA เพราะ usability ช่วงภัยพิบัติ
  • ช่องทางแจ้ง tracking_token ซ้ำเมื่อ donor ทำหาย (SMS resend? ผูกกับเบอร์?) — ยังไม่กำหนด
  • รายละเอียด field ของ donation / campaign / slot doc → canonical ที่ schema.md §2.3 (donation schema_v 2, CR-005 §F) · §2.4 (donation_campaign) · §2.13 (donation_slot, DN-5)

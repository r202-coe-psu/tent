Flow Logic

  • Registration Start เปิด form ใน shelter scope ปัจจุบันเท่านั้น
  • Registration Form ต้องแยก required, optional, sensitive, and future follow-up fields อย่างชัดเจน
  • submit ครั้งแรกทำ schema validation ก่อน, จากนั้นจึงเรียก duplicate hint path ถ้ามี match ที่ threshold ถึงเกณฑ์
  • Duplicate hint เป็น soft stop: user ต้องสามารถ inspect candidate และกลับมาส่ง record ใหม่ได้ โดย override reason ถูก audit
  • submit สำเร็จต้องสร้าง `evacuees` record, `shelter_evacuee_code`, `current_stay.status=registered`, และ registration metadata เพื่อรองรับ offline/fallback trace ในอนาคต
  • Result screen ต้องเสนอ next action เร็ว: `Add Screening`, `Open Profile`, `Proceed to Check-in`

API and Data Intent

  • Primary API group: `POST /evacuees`, `PATCH /evacuees/{id}`, `GET /evacuees/{id}`
  • Required create payload must at least support `shelter_id`, `identity.name`, `identity.gender`, and registration source metadata
  • Optional create/edit fields may include phone, emergency contact, birth year, religion, `privacy.search_excluded` and masked/hashed identity derivatives as policy allows
  • Duplicate hint can be inline from create response or separate preflight search endpoint, but override reason must be auditable
  • Normal APIs must never expose `identity.national_id_hash`

Acceptance Criteria

  • Registration Officer creates a person without creating Household
  • Minimum required path completes within the intended fast-lane flow
  • Duplicate hint does not hard-block progress and override reason is retained
  • Created record is immediately searchable within shelter scope
  • Person Profile reflects newly created registration state and next-action handoff

Open Questions

  • Field conflict: PRD says `name + gender` required while data dictionary still marks `identity.gender` optional
  • Duplicate hint threshold and comparison fields still need API-contract precision
  • Optional/sensitive field matrix still needs PO/PM sign-off before final frontend labeling

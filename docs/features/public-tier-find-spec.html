Summary

หน้า "ระบบสืบค้นญาติและครอบครัว" (RESTORING FAMILY LINKS) — nav link "สืบค้นญาติ" ของ Public Portal (landing hand off มา ที่ /search). เป็นจุดที่ ญาติ/ครอบครัวผู้ประสบภัย ค้นหาบุคคลและ ยืนยันว่าเขาปลอดภัยและอยู่ที่ศูนย์ใด เพื่อบรรเทาความเครียดโดยไม่ต้องออกเดินทางตามหา. นี่คือ FAM tier ของ public/no-login service (ดู Public Tier Flow Spec §FAM).

หน้านี้เป็น Public / No-Login Tier (FAM) — ค้นหาได้โดยไม่ login. response แสดง ข้อมูลส่วนบุคคลแบบ จำกัด + Data Masking ตาม PDPA; ทุก query ถูก audit. อ่าน central-only — trust boundary ดู Public Tier Flow Spec
เป้าหมายหลัก: หาผู้ประสบภัยให้เจอและรู้ว่าปลอดภัยอยู่ที่ไหน — usability มาก่อน friction ช่วงภัยพิบัติ. ออกแบบให้ใช้บนมือถือใน network ไม่เสถียร; รองรับ TH/EN (EN toggle)
Change tracking: หน้านี้และ data-exposure decisions ของมัน tracked ใน CR-005 §E. การค้นด้วยเลขบัตร 13 หลัก + ชุด field ที่คืน (FS-1/FS-2) เป็นการ ขยาย exposure surface ของ FAM tier เกินร่างเดิมใน public-tier-flow-spec — approved 2026-06-22. ดูส่วน Data Exposure & Privacy

3 · Search Methods & Matching Rules (FS-1)

หนึ่งช่องค้น — ระบบ auto-detect ประเภท input จากรูปแบบ แล้ว match ตามกติกาด้านล่าง. ค้นได้จากทุกช่องทางนี้ (เป้าหมาย: หาผู้ประสบภัยให้เจอ):

Input Detect rule Match Anti-enumeration
เลขบัตรประชาชน 13 หลัก ตัวเลข 13 ตัว (ตัดขีด/เว้นวรรค) exact match ครบ 13 หลักเท่านั้น ไม่รับ partial; rate-limit + audit
หมายเลขหนังสือเดินทาง (passport) alphanumeric รูปแบบ passport exact match เต็มเลขเท่านั้น ไม่รับ partial; rate-limit + audit
เบอร์โทรศัพท์ ตัวเลขรูปแบบเบอร์ไทย (normalize) exact match ครบทั้งเบอร์เท่านั้น ไม่รับ partial; rate-limit + audit
ชื่อ / นามสกุล ข้อความที่ไม่เข้ารูปแบบเลขข้างบน token match ขั้นต่ำ 3 ตัวอักษร (match ชื่อ หรือ สกุล หรือทั้งคู่) max 10 ผล/query; rate-limit แรง; PoW เมื่อ pattern ผิดปกติ

Matching Behavior

  • Exact-match identifiers (เลขบัตร / passport / เบอร์) — ต้องครบเต็มเสมอเพื่อกัน enumeration; ไม่มี endpoint list-all / prefix search
  • Consent model = opt-out (เหมือน FAM): ถ้า privacy.search_excluded = true → hide ทั้ง record แม้ query ตรงตัว (เลขบัตร/เบอร์ ตรงเป๊ะก็ไม่ขึ้น)
  • ไม่พบ → ตอบ generic empty (ไม่บอกว่า "มีแต่ถูกซ่อน") กัน probing
  • ทุก query (ยกเว้น "จำลอง") เขียน search_audit ที่ central: query type, hash ของ query, IP hash, จำนวนผลลัพธ์

5 · Masking Rules (FS-3)

ชื่อ-นามสกุล — ชื่อจริงเต็ม + สกุล mask

ตัดสินใจตาม FAM spec เดิม (override mockup ที่ mask ทั้งสองส่วน): แสดง ชื่อจริงเต็ม เพื่อให้ญาติยืนยัน ตัวบุคคลได้, mask เฉพาะ นามสกุล. นับเป็น grapheme cluster (สระ/วรรณยุกต์ไทยเกาะพยัญชนะ):

  • สกุลยาว ≥ 5 ตัว: เก็บ 2 ตัวแรก + **** + 2 ตัวท้าย — "รัตนโกสินทร์" → "รั****นทร์" (ตัวอย่างกติกา)
  • สกุลยาว ≤ 4 ตัว: เก็บตัวแรกตัวเดียว + **** — "แสง" → "แ****"
  • **** ความยาวคงที่เสมอ — ไม่เปิดเผยความยาวจริงของนามสกุล

เลขบัตรประชาชน — แสดง 3 หน้า + 3 ท้าย

  • format คงที่ NNN-XXXX-XX-NNN — เก็บ 3 หลักแรก + 3 หลักท้าย, mask ตรงกลางทั้งหมด (เช่น 390-XXXX-XX-192)
  • ไม่คืนเลขบัตรเต็มในทุก response

ภูมิลำเนาเดิม — mask เลขที่บ้าน

  • mask บ้านเลขที่ (**/*) — แสดงเฉพาะระดับ ถนน/ตำบล/อำเภอ/จังหวัด (พอให้ทราบพื้นที่ ไม่ระบุพิกัดบ้านชัด)

6 · Data Exposure & Privacy (FAM tier)

หน้านี้ขยาย exposure surface ของ FAM tier เกิน ร่างเดิมใน public-tier-flow-spec §FAM (ซึ่งจำกัดที่ ชื่อ/เบอร์ + ห้ามเลขบัตร/field อื่น). การขยายนี้ approved 2026-06-22 tracked ใน CR-005 §E
Surface Field สถานะ exposure
ค้นด้วยเลขบัตร 13 หลัก / passport query (exact match) APPROVED FS-1 เพิ่มช่องทางค้น (exact-only กัน enumeration) — เกิน FAM เดิม
ค้นด้วยเบอร์โทร / ชื่อ-สกุล query OPEN ได้ปกติ ตาม FAM เดิม (เบอร์ exact, ชื่อ ≥3 ตัว)
ชื่อจริง + สกุล (mask) name OPEN ได้ปกติ ชื่อเต็ม + สกุล mask (FS-3)
เลขบัตร (mask 3+3) / เพศ national_id (masked) / gender APPROVED FS-2 เกิน FAM เดิม — approved เพื่อยืนยันตัวบุคคล
ชื่อศูนย์ / โซนความดูแล shelter_name / care_zone APPROVED FS-2 โซน = ชื่อโซนเท่านั้น (ไม่ลง medical detail)
ภูมิลำเนาเดิม (mask) / เวลาลงทะเบียน origin_address (masked) / checked_in_at APPROVED FS-2 address mask เลขที่บ้าน
สถานะความปลอดภัย status OPEN ได้ปกติ in_shelter / moved / checked_out (core ของหน้านี้)
เบอร์โทร / เลขบัตรเต็ม / medical detail — ห้ามคืน ไม่ปรากฏใน response ใด ๆ (แม้ query ด้วยเบอร์)

หลักการ (เหมือน FAM tier)

  • เห็นน้อยที่สุดเท่าที่จำเป็น — projection เลือก field ที่อนุญาตก่อนตอบ
  • opt-out: search_excluded → hide ทั้ง record
  • exact-only สำหรับ identifier (เลขบัตร/passport/เบอร์) กัน enumeration
  • audit ทุก query; rate-limit ต่อ IP; PoW เมื่อ pattern ผิดปกติ
  • อ่าน central-only — ไม่ fallback ไป edge/LAN

Accessibility & Performance Requirements

  • WCAG 2.1 AA — contrast ≥ 4.5:1; PDPA notice + status badge ต้องอ่านได้
  • search input มี aria-label; ผลค้นหาประกาศผ่าน aria-live="polite" ("พบ N รายการ")
  • ปุ่ม "ค้นหา"/"จำลอง" เป็น <button> focusable
  • status badge ไม่สื่อด้วยสีอย่างเดียว — มี icon + ข้อความกำกับ
  • LCP ≤ 2.5 s บน 3G; ผลค้นหา fetch แยกจาก paint ของ shell
  • ภาษาไทย: lang="th" บน root เมื่อ locale = TH

UAT Checklist

  • ☐ เปิด /search โดยไม่ login → เห็น hero + search card + PDPA notice
  • ☐ ค้นด้วยเลขบัตร 13 หลักเป๊ะ → พบ record; เลขบัตรไม่ครบ → validation error (ไม่ยิง API)
  • ☐ ค้นด้วย passport / เบอร์โทร (เต็ม) → พบ record; partial → ไม่พบ
  • ☐ ค้นด้วยชื่อ/นามสกุล (≥3 ตัว) → match list (max 10)
  • ☐ result card แสดง: ชื่อเต็ม + สกุล mask, สถานะปลอดภัย, เลขบัตร mask (3+3), เพศ, ศูนย์, ภูมิลำเนา mask, เวลาลงทะเบียน, โซน
  • ☐ response ไม่มี เบอร์โทร / เลขบัตรเต็ม / medical detail
  • ☐ record ที่ search_excluded → ไม่ขึ้นแม้ค้นเลขบัตรตรงเป๊ะ
  • ☐ ค้นไม่พบ → empty state generic (ไม่บอกว่ามีแต่ถูกซ่อน)
  • ☐ ค้นรัว ๆ เกิน limit → RATE_LIMITED banner
  • ☐ ทุกการค้นเขียน search_audit ที่ central
  • ☐ ไม่มีปุ่ม "จำลอง"/demo บนหน้า (FS-4 ตัดออก)
  • ☐ ตัด central → ค้นไม่ได้ตอบ error ที่ควบคุมได้ (ไม่ fallback edge)
  • ☐ deep-link /search?q=... จาก landing → ค้นอัตโนมัติ

Screen Flow

  • User enters from Profile Summary after registration or search
  • Screening Form must preserve current demographic and stay context while editing screening fields
  • Fast track may come from explicit screening triggers or manual override; both need source/reason stored
  • Medical Details Panel renders full fields only to medical roles; non-medical roles see summary chips or hidden placeholders
  • Save success returns updated track state and dashboard-impact hint, not raw dashboard recomputation in UI

API and Data Intent

  • Likely API groups: `PATCH /evacuees/{id}`, dedicated screening subresource, or structured patch semantics in API contract
  • Data maps to `evacuees.screening`, `evacuees.medical_profile`, and related vulnerability/food/care fields from data dictionary
  • Save response must return role-safe summary shape for immediate profile refresh
  • Audit events required for screening update, medical update, and track override
  • Dashboard downstream depends on confirmed vulnerability and fast-track counts only

Acceptance Criteria

  • Medical role records elderly or mobility issue and person becomes fast track when configured trigger applies
  • Non-medical role cannot see raw medical note, allergy, medication or national-ID-derived internals by direct API call or UI
  • Manual fast-track override requires reason and is auditable
  • Updated screening summary is visible on profile and reflected in dashboard aggregate paths

Open Questions

  • Screening field list and sensitive-field matrix still need stakeholder sign-off
  • Manager-level visibility into medical summaries needs final permission-matrix decision
  • Trigger rules for fast-track should be fixed in API contract or shared rule definition

Flow Rules

  • ID/QR Result appears immediately after successful registration and persists on profile
  • Search must respect shelter scope and role masking even when query is exact code/QR
  • QR scan path resolves code server-side, then routes to profile or direct action surface depending on permission
  • Check-in path must reject duplicate occupancy inflation and still allow over-capacity admission as warning-only
  • Check-out path requires person to be in a checked-in state; invalid state returns controlled failure
  • History is append-only evidence for movement actions and must remain consistent with current stay summary

API and Data Intent

  • Search/QR endpoints: `GET /evacuees/search`, `GET /qr/{code}` or equivalent resolver
  • Movement endpoints: `POST /evacuees/{id}/check-in`, `POST /evacuees/{id}/check-out`
  • Movement requests require idempotency key so retries do not inflate occupancy
  • Data sources: `evacuees.current_stay`, `movement_events`, `id_counters`, `audit_logs`
  • Check-in/out responses should include refreshed stay summary and event identifier for UI reconciliation

Acceptance Criteria

  • Registered person can be found via Shelter ID/QR within shelter scope
  • Unauthorized role cannot use search or QR resolve to view restricted medical/sensitive detail
  • Check-in increments occupancy correctly and duplicate retry with same idempotency key does not create a second increment
  • Check-out from a non-checked-in state fails clearly without corrupting occupancy
  • Movement history remains inspectable after check-out and re-check-in

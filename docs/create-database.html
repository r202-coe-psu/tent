CouchDB Lab — สร้าง Database

ขั้นตอนสร้าง Database ใน CouchDB

How-to CouchDB 3.5 2026-06-10

คู่มือทีละขั้น ตาม pattern จริงในโปรเจคนี้ — สร้าง DB → ตั้งสิทธิ์ → design doc → user → grant access

ภาพรวม

CouchDB เป็น REST API ล้วน — สร้าง database = ส่ง HTTP request ไม่มี schema ไม่ต้อง migration ทุกอย่างคุยผ่าน HTTP method (PUT / GET / DELETE) ตรงไปยัง URL ของ resource

หลักสำคัญ: ทุก privileged call ต้องผ่าน admin client ฝั่ง server (web/src/lib/server/couch-admin.ts) เพื่อไม่ให้ admin password หลุดไปฝั่ง browser โค้ดในเอกสารนี้อ้างอิงจาก api/register/+server.ts และ api/admin/demo/+server.ts

ก่อนเริ่ม — Admin Client

admin URL พร้อม credential เก็บใน private env var COUCHDB_ADMIN_URL ไม่ถูก bundle ไป browser มี helper 2 ตัว:

Helperพฤติกรรมใช้ตอน
adminRaw(path, method, body?)คืน { status, data } ไม่ throw แม้ HTTP errorต้องเช็ค status เอง (เช่น 412 = มีอยู่แล้ว)
adminFetch<T>(path, init?)parse JSON แล้ว throw reason ของ CouchDB ถ้า errorอยากให้ fail ทันทีเมื่อพลาด
 
// ทุก step ข้างล่างเรียกผ่านสองตัวนี้
import { adminRaw, adminFetch } from '$lib/server/couch-admin';

ภาพรวม Flow

ลำดับการ setup ทั้งระบบ — จากสร้าง DB จนพร้อมใช้งาน (อ้างอิง setupDemo())

1สร้าง Database

PUT /{dbname} — แค่นี้ database ก็เกิด

const { status, data } = await adminRaw('/notes', 'PUT');
// status 201 → สร้างสำเร็จ
// status 412 → มีอยู่แล้ว (Precondition Failed) → ข้ามได้
// อื่นๆ        → error, ดู data.reason
ชื่อ database: ตัวพิมพ์เล็ก เริ่มด้วยตัวอักษร ใช้ได้แค่ a-z 0-9 _ $ ( ) + - / ชื่อขึ้นต้น _ สงวนไว้ให้ระบบ (เช่น _users, _replicator)

2ตั้ง _security — ใครเข้าได้

PUT /{dbname}/_security — database เปล่าใครก็เข้าได้ ต้องตั้ง _security เพื่อจำกัดสิทธิ์ 

const sec = {
  admins:  { names: [], roles: [] },
  members: { names: [], roles: ['team:alpha'] }
};
await adminRaw('/demo_alpha/_security', 'PUT', sec);
  • members.names — รายชื่อ user ที่เข้าได้ (รายคน)
  • members.roles — role ที่เข้าได้ (เช่น team:alpha)
  • admins — ใครแก้ design doc / security ของ DB นี้ได้
  • ปล่อยว่างทั้งหมด → ใครก็เข้าได้
ระวัง: ชื่อที่ไม่อยู่ใน members โดน CouchDB reject 403 ทันทีตั้งแต่ชั้น API — ก่อนถึง frontend ด้วยซ้ำ

3Design Doc — Validation (optional)

PUT /{dbname}/_design/{name} — ใส่ validate_doc_update เพื่อบังคับกฎตอนเขียนทุก doc (เช่น กันเขียนข้าม team) 

const validateFn = `function(newDoc, oldDoc, userCtx) {
  if (userCtx.roles.indexOf('_admin') !== -1) return;
  var team = newDoc.team;
  if (!team) throw({ forbidden: 'team field required' });
  if (userCtx.roles.indexOf('team:' + team) === -1)
    throw({ forbidden: 'Access denied: requires role team:' + team });
}`;

// update ต้องแนบ _rev เดิม → GET ก่อน
const { data: existing } = await adminRaw('/demo_shared/_design/access', 'GET');
const rev = existing?._rev;
await adminRaw('/demo_shared/_design/access', 'PUT', {
  _id: '_design/access',
  ...(rev ? { _rev: rev } : {}),
  validate_doc_update: validateFn
});
เมื่อไหร่ใช้: ข้ามได้ถ้าคุม access ระดับ DB ด้วย _security พอแล้ว ใช้ design doc เมื่อต้องการกฎ ภายใน DB (เช่น 1 DB หลาย team)

4สร้าง User

user เก็บใน database พิเศษ _users — doc id ต้องขึ้นต้น org.couchdb.user: 

await adminFetch(
  `/_users/org.couchdb.user:${encodeURIComponent(username)}`,
  {
    method: 'PUT',
    body: JSON.stringify({
      name: username,
      password,           // CouchDB hash ให้เอง
      roles: [],          // เช่น ['team:alpha']
      type: 'user'        // ต้องมี
    })
  }
);
ต้องครบ: name, type: 'user', และ doc id ที่ขึ้นต้น org.couchdb.user: ขาดอย่างใดอย่างหนึ่ง CouchDB reject

5Grant Access — ให้ User เข้า DB

เพิ่ม user เข้า DB ที่มีอยู่แล้ว = อ่าน _security เดิม → push ชื่อ → PUT กลับ (อ้างอิง grantNotesAccess()) 

async function grantNotesAccess(username) {
  const security = await adminFetch('/notes/_security');
  security.members ??= { names: [], roles: [] };
  security.members.names ??= [];
  if (!security.members.names.includes(username)) {
    security.members.names.push(username);
  }
  await adminFetch('/notes/_security', {
    method: 'PUT',
    body: JSON.stringify(security)
  });
}
Pattern read-modify-write: อ่านของเดิมมาก่อนเสมอ แล้วแก้เฉพาะ field ที่ต้องการ — ไม่งั้นจะเขียนทับ members เดิมหาย

เรื่อง _rev — กฎเหล็ก

CouchDB ใช้ MVCC — ทุก document มี _rev การ แก้ หรือ ลบ doc ที่มีอยู่แล้ว ต้องแนบ _rev ล่าสุด ไม่งั้นได้ 409 Conflict 

// แก้ doc → GET เอา _rev ก่อนเสมอ
const { data } = await adminRaw('/db/doc-id', 'GET');
await adminRaw('/db/doc-id', 'PUT', { ...body, _rev: data._rev });

// ลบ doc → ส่ง rev เป็น query param
await adminRaw(`/db/doc-id?rev=${data._rev}`, 'DELETE');
สร้างใหม่ vs แก้: สร้าง doc ครั้งแรก = ไม่ต้องมี _rev · แก้ของเดิม = ต้องมี _rev — โค้ดในโปรเจค GET ก่อนทุกครั้งเพื่อรองรับทั้งสองกรณี (rev ? { _rev: rev } : {})

Status Codes ที่เจอบ่อย

Codeความหมายทำยังไง
201สร้าง DB / doc สำเร็จผ่าน
200อ่าน / อัปเดตสำเร็จผ่าน
412DB มีอยู่แล้ว (สร้างซ้ำ)ข้ามได้ — ถือว่า idempotent
409Conflict — _rev ไม่ตรง/ขาดGET เอา _rev ล่าสุดมาใหม่
401ไม่ได้ authเช็ค COUCHDB_ADMIN_URL / cookie
403ไม่มีสิทธิ์ (_security กัน)เพิ่มชื่อใน members หรือใช้ admin

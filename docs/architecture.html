RBAC ปัจจุบัน

ระบบไม่ได้สร้าง RBAC เอง — CouchDB จัดการ roles ทั้งหมด แต่ละ user มี roles[] ใน _users DB และแต่ละ database มี _security doc คุมว่าใครเข้าได้

Userrolesสิทธิ์
admin["_admin"]ทุกอย่าง — อ่าน/เขียน/ลบ ทุก DB
demo[]เฉพาะ DB ที่ถูก whitelist ใน _security.members

Notes DB — การป้องกันระดับ database

// GET /notes/_security
{
  "admins":  { "names": [], "roles": [] },
  "members": { "names": ["demo", "alice"], "roles": [] }
}

ชื่อไม่อยู่ใน members.names → CouchDB reject ทันที (403) ก่อน ถึง frontend ตอน register/create user เราเรียก grantNotesAccess() เพื่อเพิ่มชื่อเข้า list นี้

Shelter Management — Offline-First (อุทกภัย)

สถานการณ์: น้ำท่วม ไม่มี internet แต่มีไฟ บางศูนย์มี edge computer บางศูนย์ไม่มี ระบบซับซ้อน — รับคนเข้า, บริจาค, จอง, แผนที่อพยพ, ครัว, คลังสินค้า นี่คือ killer use case ของ CouchDB เพราะถูกออกแบบมาเพื่อ offline-first multi-master replication โดยตรง

ความต้องการCouchDB ให้อะไร
ไม่มี internetreplication เป็น async ทำงาน local 100%
edge บางที่ไม่มีPouchDB ในมือถือ = CouchDB ย่อส่วน sync กันเองได้
หลายคนแก้ offline พร้อมกันMVCC + revision tree, merge ตอน sync
ไฟดับ/เครื่องพังข้อมูลกระจายหลาย replica ไม่หายทั้งหมด
เน็ตกลับมาsync ต่อจาก checkpoint อัตโนมัติ

Conflict & Event-Sourcing

Offline หลายจุดแก้พร้อมกัน → CouchDB ไม่ทิ้งข้อมูล เก็บทุก revision แล้วเลือก winner แบบ deterministic (ไม่ใช่ semantic) เราต้อง resolve conflict ที่ "สำคัญ" เอง

ระบบconflict ที่เกิดวิธีแก้
intake2 ศูนย์ลงทะเบียนคนเดิมใช้ national_id เป็น _id, dedupe ตอน merge
warehouse2 คนเบิกของชิ้นเดียว offlineเก็บเป็น event log (−1 / +1) ไม่เก็บยอดคงเหลือตรงๆ
reservationจองเตียงเดียวกัน 2 คนresolve ตอน sync, แจ้ง staff ตัดสิน
กฎทอง: ระบบที่นับจำนวน (stock, เตียง, อาหาร) → ออกแบบเป็น append-only event/ledger ไม่ใช่ mutable counter → ไม่มี conflict เพราะไม่มีใครแก้ field เดียวกัน ยอดคงเหลือ = sum ของ events

สรุปคำแนะนำ

  • ใช้ Pattern 1 (DB-per-subscope + roles) เป็นฐาน
  • Topology: Cloud → Edge → Device (hub-and-spoke)
  • Edge ข้างเคียง sync กันผ่าน LAN → redundancy
  • Shelter ไม่มี edge → ใช้ tablet เป็น soft-edge hub
  • ระบบนับจำนวน → event-sourcing/ledger กัน conflict
  • Filtered replication → device โหลดเฉพาะ scope ตัวเอง
  • _id ใช้ natural key (เช่น national_id) กัน duplicate
จุดที่ต้องลงแรงออกแบบมากสุด = conflict resolution ของ warehouse / kitchen / reservation เพราะเป็น mutable state ที่หลายจุดแย่งแก้ตอน offline — ที่เหลือ CouchDB จัดการให้เกือบหมด

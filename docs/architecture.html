CouchDB Lab — RBAC & Offline-First Architecture

Architecture RBAC Offline-First 2026-06-10

สรุปสถาปัตยกรรม: ระบบสิทธิ์ปัจจุบัน → multi-database scoping → ระบบ shelter management ที่ทำงาน offline ได้

ภาพรวม

ระบบนี้ใช้ Central CouchDB เป็น source of truth ทั้งเรื่อง authentication และ authorization โดย frontend (SvelteKit + PouchDB) ทำหน้าที่ อ่าน สิทธิ์มาแสดงผลและ gate route เท่านั้น ความปลอดภัยจริงบังคับใช้ที่ชั้น Central CouchDB; Edge fallback ใช้ _session แยกและไม่ให้สิทธิ์แทน Central API

RBAC ปัจจุบัน

ระบบไม่ได้สร้าง RBAC เอง — CouchDB จัดการ roles ทั้งหมด แต่ละ user มี roles[] ใน _users DB และแต่ละ database มี _security doc คุมว่าใครเข้าได้

Userrolesสิทธิ์
admin["_admin"]ทุกอย่าง — อ่าน/เขียน/ลบ ทุก DB
demo[]เฉพาะ DB ที่ถูก whitelist ใน _security.members

Notes DB — การป้องกันระดับ database

// GET /notes/_security
{
  "admins":  { "names": [], "roles": [] },
  "members": { "names": ["demo", "alice"], "roles": [] }
}

ชื่อไม่อยู่ใน members.names → CouchDB reject ทันที (403) ก่อน ถึง frontend ตอน register/create user เราเรียก grantNotesAccess() เพื่อเพิ่มชื่อเข้า list นี้

Auth Flow

ข้อควรระวัง: guard ฝั่ง frontend = UX convenience เท่านั้น ถ้าแก้ authStore.user.roles ใน DevTools จะเข้าหน้า admin ได้ — แต่ CouchDB ยัง block API call จริงถ้า session ไม่ใช่ _admin security จริงอยู่ที่ CouchDB

Multi-DB Scoping — user เข้าได้เฉพาะบาง database

Use case: ร้านอาหารหลายสาขา นาย A สาขา 1 เข้าได้แค่ stock ไม่เข้า bill วิธีที่ scale ดีสุดคือใช้ roles แทน names แล้วตั้งชื่อแบบ branch{id}:{resource} 

// _users documents
{ "name": "nai_a", "roles": ["branch1:stock"] }
{ "name": "nai_b", "roles": ["branch1:stock", "branch1:bill", "branch1:manager"] }
{ "name": "nai_c", "roles": ["branch2:stock"] }

// แต่ละ DB ตั้ง _security ด้วย roles
branch1_stock  → members.roles: ["branch1:stock"]
branch1_bill   → members.roles: ["branch1:bill", "branch1:manager"]
branch2_stock  → members.roles: ["branch2:stock"]

Shelter Management — Offline-First (อุทกภัย)

สถานการณ์: น้ำท่วม internet/WAN ไม่เสถียร แต่มีไฟ บางศูนย์มี LAN Edge fallback บางศูนย์ไม่มี ระบบซับซ้อน — รับคนเข้า, บริจาค, จอง, แผนที่อพยพ, ครัว, คลังสินค้า นี่คือ killer use case ของ CouchDB/PouchDB เพราะ client เขียน local ก่อนเสมอ แล้วเลือก remote sync ได้แบบควบคุม: Central เป็น remote ปกติ, LAN Edge ใช้เฉพาะตอน WAN/Central ล่ม

ความต้องการCouchDB ให้อะไร
ไม่มี internetPouchDB ทำงาน local 100% แล้ว sync backlog เมื่อ remote กลับมา
edge บางที่ไม่มีไม่มี edge = local-only ระหว่าง outage; มี edge = fallback replica ผ่าน LAN เท่านั้น
หลายคนแก้ offline พร้อมกันMVCC + revision tree, merge ตอน sync
ไฟดับ/เครื่องพังข้อมูลกระจายหลาย replica ไม่หายทั้งหมด
เน็ตกลับมาclient failback ไป Central; Edge sync backlog ขึ้น Central จาก checkpoint

Replication Topology

Client เขียน local PouchDB ก่อนเสมอ และมี active remote ได้ครั้งละหนึ่งเป้าหมาย: Central เมื่อ reachable, LAN Edge เฉพาะตอน WAN/Central ล่ม, local-only เมื่อไม่มี remote ใดใช้ได้

  • ห้าม device รัน long-lived replication ไป Central และ Edge พร้อมกัน — remote target ต้อง switch แบบ exclusive
  • Central และ Edge มี _session cookie แยกกัน; session fallback ที่ Edge ใช้เรียก Central service API ไม่ได้
  • Edge เป็น LAN continuity/fallback replica และ sync backlog ขึ้น Central เมื่อ WAN กลับมา ไม่ใช่ normal client hub
  • Public/no-login endpoints อยู่ที่ Central เท่านั้นและไม่ fallback ไป Edge

Database layout

ต่อ shelter หนึ่งศูนย์:
  shelter{id}_intake       ระบบรับคนเข้า
  shelter{id}_donations    บริจาคของ
  shelter{id}_reservation  ระบบจอง
  shelter{id}_kitchen      ครัว
  shelter{id}_warehouse    คลังสินค้า

ใช้ร่วมทุกศูนย์ (push จาก central):
  shared_evacuation_map    แผนที่อพยพ
  shared_shelter_registry  ทะเบียนศูนย์ + capacity

แยก DB ตาม subsystem → sync เลือกเฉพาะที่จำเป็น, คุม role แยก, DB นึงพังไม่ลามทั้งหมด

ศูนย์ไม่มี LAN Edge

Default ของศูนย์ไม่มี Edge คือ Central-first เมื่อ online และ local-only ระหว่าง outage ถ้าต้องมี soft-edge ให้กำหนดเป็น fallback replica ชั่วคราวเท่านั้น ไม่ใช่ normal hub และไม่ให้ session ของ soft-edge ใช้แทน Central API ได้

Conflict & Event-Sourcing

Offline หลายจุดแก้พร้อมกัน → CouchDB ไม่ทิ้งข้อมูล เก็บทุก revision แล้วเลือก winner แบบ deterministic (ไม่ใช่ semantic) เราต้อง resolve conflict ที่ "สำคัญ" เอง

ระบบconflict ที่เกิดวิธีแก้
intake2 ศูนย์ลงทะเบียนคนเดิมใช้ national_id เป็น _id, dedupe ตอน merge
warehouse2 คนเบิกของชิ้นเดียว offlineเก็บเป็น event log (−1 / +1) ไม่เก็บยอดคงเหลือตรงๆ
reservationจองเตียงเดียวกัน 2 คนresolve ตอน sync, แจ้ง staff ตัดสิน
กฎทอง: ระบบที่นับจำนวน (stock, เตียง, อาหาร) → ออกแบบเป็น append-only event/ledger ไม่ใช่ mutable counter → ไม่มี conflict เพราะไม่มีใครแก้ field เดียวกัน ยอดคงเหลือ = sum ของ events

สรุปคำแนะนำ

  • ใช้ Pattern 1 (DB-per-subscope + roles) เป็นฐาน
  • Topology: Central-first + LAN Edge fallback (ไม่ใช่ Edge เป็น normal hub)
  • Device เขียน local PouchDB ก่อน แล้ว sync ไป active remote เพียงหนึ่งตัว: Central, Edge fallback, หรือ local-only
  • Edge เก็บ fallback replica และ sync backlog ขึ้น Central เมื่อ WAN กลับมา
  • Shelter ไม่มี edge → Central เมื่อ online, local-only ตอน outage; soft-edge ใช้ได้เฉพาะ fallback ที่กำหนดชัด
  • ระบบนับจำนวน → event-sourcing/ledger กัน conflict
  • Filtered replication → device โหลดเฉพาะ scope ตัวเอง
  • _id ใช้ natural key (เช่น national_id) กัน duplicate
จุดที่ต้องลงแรงออกแบบมากสุด = conflict resolution ของ warehouse / kitchen / reservation เพราะเป็น mutable state ที่หลายจุดแย่งแก้ตอน offline — ที่เหลือ CouchDB จัดการให้เกือบหมด
